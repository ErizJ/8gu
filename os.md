# 第一章 操作系统概述

## 1.1 什么是操作系统 ★★★

### 操作系统定义

操作系统（Operating System，简称 OS）是管理计算机硬件与软件资源的**系统软件**，也是计算机系统的内核与基石。它运行在**内核态**，为应用程序提供执行环境。

- **向下**：管理硬件资源（CPU、内存、磁盘、网卡、外设）
- **向上**：为用户/应用程序提供服务接口（系统调用、Shell）

### 操作系统作用

| 功能 | 说明 |
|------|------|
| **进程管理** | 进程的创建、调度、终止，以及进程间同步与通信 |
| **内存管理** | 内存的分配与回收，虚拟内存，地址映射 |
| **文件管理** | 文件的创建、读写、删除，目录结构管理 |
| **设备管理** | 管理 I/O 设备，提供统一的设备驱动接口 |
| **资源管理** | 对 CPU、内存、磁盘、网络等资源进行统一分配调度 |

### 面试题

**Q: 什么是操作系统？**

> 操作系统是管理计算机硬件与软件资源的系统软件。它运行在内核态，负责进程管理、内存管理、文件管理和设备管理，并为应用程序提供执行环境和编程接口（系统调用）。核心目标是：**方便用户使用** + **高效利用硬件资源**。

**Q: 操作系统有哪些功能？**

> 五大功能：进程管理（调度、同步、通信）、内存管理（分配回收、虚拟内存）、文件管理（文件系统、目录）、设备管理（驱动、I/O 控制）、用户接口（系统调用、Shell）。

## 1.2 用户态与内核态 ★★★★★

### 用户态（User Mode）

- CPU 处于较低的**特权级别（Ring 3）**
- 只能执行**非特权指令**（如算术运算、逻辑运算）
- 不能直接访问硬件资源或关键内核数据结构
- 应用程序、库函数运行在此模式下

### 内核态（Kernel Mode）

- CPU 处于最高的**特权级别（Ring 0）**
- 可以执行**特权指令**（如修改页表、开关中断、I/O 操作）
- 可以访问所有内存地址空间
- 操作系统内核代码运行在此模式下

### 特权指令

只能在内核态执行的指令：
- 修改 CPU 状态寄存器（CR0~CR4）
- 设置中断向量表
- 修改页表基址寄存器
- 执行 I/O 指令（in/out）
- 停机指令（hlt）
- 清除内存（invlpg）

### 非特权指令

用户态可执行的指令：算术运算、逻辑运算、数据传送（mov）、跳转、函数调用等。

### 面试题

**Q: 用户态和内核态有什么区别？**

> 1. **特权级别不同**：用户态在 CPU Ring 3，内核态在 Ring 0
> 2. **权限不同**：用户态不能执行特权指令（如 I/O 操作、修改页表），内核态可以
> 3. **地址空间不同**：用户态只能访问自己的虚拟地址空间，内核态可以访问全部物理内存
> 4. **切换方式**：用户态通过**系统调用、中断、异常**三种方式陷入内核态
> 5. **设计目的**：保护系统安全 —— 用户程序崩溃不会影响内核

**Q: 为什么要区分用户态和内核态？**

> **安全性 + 稳定性**：
> - 防止用户程序直接操作硬件造成系统崩溃
> - 防止用户程序恶意访问其他进程的内存
> - 通过内核统一管理资源，避免冲突
> - 实现**隔离**：一个用户程序 crash 不会导致整个系统 crash

## 1.3 系统调用 ★★★★★

### 什么是系统调用

系统调用（System Call）是操作系统提供给应用程序的**编程接口**，是用户态程序请求内核服务的**唯一入口**。应用程序通过系统调用陷入内核态，由内核代为执行特权操作。

**流程：**
```
用户程序 → 系统调用 → 陷入内核(trap) → 内核执行 → 返回用户态
```

以 `write(fd, buf, count)` 为例：
1. 用户程序调用 glibc 封装的 `write()`
2. glibc 将系统调用号存入 `eax` 寄存器，参数存入 `ebx/ecx/edx`
3. 执行 `int 0x80`（x86 32位）或 `syscall`（x86 64位）指令
4. CPU 切换到内核态，根据系统调用号跳转到对应的内核处理函数
5. 内核执行 `sys_write()`，完成后返回用户态

### 常见系统调用

| 系统调用 | 作用 | 分类 |
|---------|------|------|
| `fork` | 创建子进程，复制父进程地址空间 | 进程控制 |
| `execve` | 加载新程序替换当前进程映像 | 进程控制 |
| `exit` | 终止当前进程 | 进程控制 |
| `wait/waitpid` | 等待子进程终止 | 进程控制 |
| `read` | 从文件描述符读取数据 | 文件操作 |
| `write` | 向文件描述符写入数据 | 文件操作 |
| `open` | 打开/创建文件 | 文件操作 |
| `close` | 关闭文件描述符 | 文件操作 |
| `mmap` | 将文件/设备映射到内存 | 内存管理 |
| `brk/sbrk` | 调整堆空间大小 | 内存管理 |
| `socket` | 创建套接字 | 网络 |
| `bind/listen/accept` | 网络服务端操作 | 网络 |
| `connect` | 建立网络连接 | 网络 |

### 面试题

**Q: 什么是系统调用？**

> 系统调用是用户程序请求操作系统内核服务的接口。用户程序不能直接访问硬件，必须通过系统调用陷入内核态，由内核代为执行。常见的如 `read`、`write`、`fork`、`open` 等。它是用户态与内核态之间的**桥梁**。

**Q: 系统调用发生了什么？**

> 1. 用户程序将**系统调用号**放入 eax/rax 寄存器，参数放入其他通用寄存器
> 2. 执行**软中断指令**（`int 0x80` 或 `syscall`）
> 3. CPU 从用户态切换到内核态：
>    - 保存用户态上下文（SS、ESP、EFLAGS、CS、EIP 压入内核栈）
>    - 加载内核栈指针
>    - 跳转到系统调用入口函数
> 4. 内核根据系统调用号查表，调用对应的处理函数
> 5. 处理完成后，调用 `iret`（32位）或 `sysret`（64位）返回用户态
> 6. 恢复用户态上下文，继续执行用户程序

### 系统调用的多种实现方式

| 指令 | 架构 | 说明 |
|------|------|------|
| `int 0x80` | x86 32位 | 传统软中断方式，较慢（需要查 IDT、权限检查） |
| `sysenter` / `sysexit` | x86 32位 | Intel 快速系统调用，比 int 快 2-3 倍 |
| `syscall` / `sysret` | x86 64位 | AMD 发起、Intel 也采用的 64 位标准方式 |

### vDSO / vsyscall（虚拟动态共享对象）

**问题**：有些系统调用不涉及任何安全敏感操作（如 `gettimeofday`），但每次都要经历 "用户态→内核态→用户态" 的切换开销（~100-200ns）。

**解决方案 — vDSO**：内核将某些**只读的系统调用**实现映射到用户态内存，用户程序直接调用，**完全避免上下文切换**。

```
普通系统调用路径：
  用户程序 → syscall 指令 → 内核态 → sysret 指令 → 用户程序
  (开销 ~100-200ns)

vDSO 路径：
  用户程序 → 直接读取内核映射到用户空间的数据 → 返回
  (开销 ~几个 ns，零上下文切换)
```

**vDSO 中的函数**：`gettimeofday`、`clock_gettime`、`time`、`getcpu` 等。

```bash
# 查看 vDSO 映射
cat /proc/self/maps | grep vdso
# 7ffc12340000-7ffc12341000 r-xp 00000000 00:00 0  [vdso]
```

### 系统调用开销详细分析

一次系统调用的开销组成（以 `syscall` 指令为例）：

| 开销来源 | 耗时 | 说明 |
|---------|------|------|
| **指令本身** | ~几十 ns | `syscall` 指令切换特权级 |
| **寄存器保存/恢复** | ~几十 ns | 保存用户态上下文到内核栈 |
| **内核处理逻辑** | 不定 | 取决于系统调用本身（如 write 可能等待 I/O） |
| **TLB 刷新** | ~几百 ns | 若有 KPTI/Meltdown 修复（见下面） |
| **Cache 污染** | 间接 | 内核代码可能踢出用户代码的缓存 |

> **总结**：空系统调用（如 `getpid`）约 **100-200ns**（不含 KPTI），有 KPTI 后增加约 30-50%。加上内核处理逻辑后时间变化很大。

## 1.4 中断与异常 ★★★★

### 中断（Interrupt）

中断是**外部硬件**产生的事件，用于通知 CPU 有事件需要处理。与当前执行的指令**异步**（不可预测）。

- **时钟中断**：定时器芯片周期性产生，用于进程调度、时间统计
- **I/O 中断**：外设（磁盘、网卡、键盘）数据传输完成时产生
- **硬中断**：由硬件设备直接产生

### 异常（Exception）

异常是 CPU **执行指令时**产生的错误或特殊条件。与当前执行的指令**同步**（可重现）。

| 类型 | 例子 | 处理方式 |
|------|------|---------|
| **故障 Fault** | 缺页异常、除零错误 | 可恢复，返回当前指令重新执行 |
| **陷阱 Trap** | 单步调试、断点(int 3) | 返回下一条指令 |
| **中止 Abort** | 硬件错误、双重故障 | 不可恢复，终止进程 |

### 时钟中断

- 由可编程间隔定时器（PIT）或 APIC 定时器周期性产生
- Linux 中默认频率 250Hz（每 4ms 一次），可配置为 100/250/300/1000Hz
- **作用**：
  - 更新系统时间（jiffies）
  - 进程时间片耗尽 → 触发调度
  - 统计进程 CPU 使用时间
  - 定时器到期处理

### I/O 中断

- 外设完成数据传输后产生，通知 CPU 数据已就绪
- 例如：网卡收到数据 → 产生中断 → CPU 执行网卡驱动的中断处理程序
- 相比**轮询**方式，中断能大幅提高 CPU 利用率

### 面试题

**Q: 中断和异常的区别？**

> | 维度 | 中断 | 异常 |
> |------|------|------|
> | **来源** | 外部硬件产生 | CPU 执行指令产生 |
> | **同步性** | 异步（与当前指令无关） | 同步（由当前指令触发） |
> | **可预测** | 不可预测 | 可重现 |
> | **处理时机** | 当前指令执行完后 | 立即处理 |
> | **例子** | 时钟中断、键盘输入、网卡收包 | 缺页、除零、断点、非法指令 |
> | **返回** | 返回下一条指令 | 故障返回当前指令，陷阱返回下一条 |

### 中断完整处理流程 ★★★★

```
硬件中断到达 CPU
    │
    ▼
1. CPU 完成当前指令 → 检查中断引脚
    │
    ▼
2. 硬件保存部分上下文（eflags, cs, eip 压栈）→ 切换到内核栈
    │
    ▼
3. 根据中断向量号查**中断向量表（IDT）** → 跳转到对应中断处理程序入口
    │
    ▼
4. 中断处理程序开始执行（内核态）
    │
    ├── 上半部（Top Half）：紧急、快速、不可中断
    │     保存全部寄存器
    │     确认中断来源（读取中断控制器状态）
    │     最紧急的数据处理（如从网卡缓冲区取数据包）
    │     调度下半部处理余下工作
    │
    └── 下半部（Bottom Half）：非紧急、可被中断
          进一步处理（如 TCP/IP 协议栈解析数据包）
          放入进程的接收队列
          实现方式：softirq、tasklet、workqueue
    │
    ▼
5. 恢复寄存器 → 执行 iret（中断返回指令）
    │
    ▼
6. CPU 回到被中断的指令继续执行（用户态）
```

**上半部 vs 下半部**：

| | 上半部 Top Half | 下半部 Bottom Half |
|------|---------------|------------------|
| **执行时机** | 中断发生立即执行 | 上半部调度，稍后执行 |
| **可否被中断** | 关中断，不可被打断 | 开中断，可被新中断打断 |
| **处理内容** | 紧急、快速的操作 | 耗时但不紧急的处理 |
| **实现** | 中断处理程序本身 | softirq、tasklet、workqueue |

> **为什么分上下半部**：如果中断处理程序执行太久（比如在中断上下文中解析完整的 TCP 包），会阻塞其他中断（甚至时钟中断），导致系统失去响应。因此将紧急部分和耗时部分分离。

### 操作系统引入中断的作用

1. **异步事件响应**：CPU 不需要轮询设备，设备就绪时通过中断通知 CPU，大幅提高 CPU 利用率
2. **进程调度的基础**：时钟中断触发内核检查时间片，是实现抢占式多任务的基础
3. **异常处理**：缺页异常、除零错误、段错误等通过异常机制处理
4. **系统调用**：用户程序通过软中断（`int 0x80` / `syscall`）主动请求内核服务
5. **实时性**：中断机制使得系统可以对紧急事件做出快速响应

## 1.5 KPTI（Kernel Page Table Isolation）★★★

### Meltdown 漏洞与 KPTI 修复

**Meltdown**（2018 年公布）利用了 CPU 的**乱序执行**特性，允许用户态程序通过侧信道（Cache timing）间接读取内核内存。传统设计下，内核空间和用户空间使用同一套页表——内核页表项在用户态不可访问（U/S 位保护），但乱序执行期间的投机访问仍能将内核数据带入 Cache。

**KPTI 修复方案**：内核态和用户态使用**两套独立的页表**。

```
传统方案（无 KPTI）：             KPTI 方案：
┌─────────────────┐              ┌─────────────────┐  ┌─────────────────┐
│ 用户 + 内核页表   │              │   用户态页表      │  │   内核态页表      │
│                 │              │  (仅有用户空间)   │  │  (用户 + 内核)   │
│ 用户空间映射      │              │  内核映射极少     │  │  完整内核映射     │
│ 内核空间映射      │              │  (仅 trampoline) │  │                 │
└─────────────────┘              └─────────────────┘  └─────────────────┘
    一套页表                         切换时切换 CR3（刷新 TLB）
```

### 性能影响

每次系统调用、中断、异常都要在用户态页表和内核态页表之间切换 → **切换 CR3 + TLB 刷新**。

| 指标 | 影响 |
|------|------|
| **系统调用开销** | 增加 30-50% |
| **上下文切换开销** | 显著增加（频繁刷新 TLB） |
| **I/O 密集型应用** | 受影响较大（大量系统调用） |
| **CPU 密集型应用** | 影响较小（系统调用少） |

**PCID（Process-Context Identifier）**：Intel 的缓解技术，给 TLB 条目打上进程 ID 标签，切换页表时不需要全刷新 TLB，减少 KPTI 的性能损失。

```bash
# 检查 KPTI 是否启用
dmesg | grep -i "page table isolation\|KPTI\|PTI"
# /proc/cpuinfo 中检查是否支持 pcid
grep pcid /proc/cpuinfo
```

### 面试题

**Q: KPTI 是什么？为什么引入？**

> KPTI（Kernel Page Table Isolation）是 Meltdown 漏洞的修复方案。传统内核和用户态共用页表，Meltdown 利用 CPU 乱序执行+侧信道从用户态读取内核数据。KPTI 将内核和用户态使用两套独立页表，系统调用时切换页表。代价：系统调用开销增加 30-50%（需切换 CR3 + TLB 刷新），PCID 可部分缓解。

# 第二章 进程与线程 ★★★★★

## 2.1 进程基础

### 什么是进程

进程（Process）是**程序的一次执行实例**，是系统进行**资源分配**和**调度**的基本单位。

```
进程 = 程序代码 + 数据 + PCB + 堆栈
```

一个程序可以对应多个进程（如多次启动 Chrome），一个进程只能对应一个程序。

### PCB（Process Control Block，进程控制块）

PCB 是操作系统用来**描述和控制进程**的数据结构，是进程在系统中的唯一标识。Linux 中用 `task_struct` 结构体表示。

**PCB 保存的内容：**

| 类别 | 内容 |
|------|------|
| **进程标识** | PID、父进程 PPID、用户 UID |
| **处理器状态** | 通用寄存器、程序计数器 PC、栈指针 SP、标志寄存器 |
| **进程调度** | 进程状态、优先级、调度策略、时间片 |
| **内存管理** | 代码段/数据段/栈段基址、页表指针 |
| **文件管理** | 打开文件描述符表、工作目录 |
| **记账信息** | CPU 使用时间、内存使用量 |
| **链接信息** | 指向父/子/兄弟进程的指针 |

### 进程状态

见 2.2 节详细分析。

### 面试题

**Q: 什么是进程？**

> 进程是**程序的一次执行**，是系统进行资源分配和调度的基本单位。它由程序代码、数据段、堆栈段、PCB 组成。PCB（进程控制块）是进程在 OS 中的唯一标识，包含了进程的所有管理信息。

**Q: PCB 中保存了什么？**

> PCB 是进程在操作系统中的"身份证"，保存了：
> 1. **标识信息**：PID、PPID、UID
> 2. **CPU 上下文**：寄存器值、PC、栈指针
> 3. **调度信息**：状态、优先级、时间片
> 4. **内存信息**：代码段/数据段/堆栈段指针、页表
> 5. **文件信息**：打开的文件描述符表
> 6. **记账信息**：CPU 时间、内存用量

## 2.2 进程状态转换 ★★★★★

### 五种状态

| 状态 | 说明 |
|------|------|
| **创建态 New** | 进程刚被创建，尚未进入就绪队列，PCB 初始化、资源分配中 |
| **就绪态 Ready** | 进程已获得除 CPU 外的所有资源，等待调度器分配 CPU |
| **运行态 Running** | 进程正在 CPU 上执行 |
| **阻塞态 Blocked** | 进程等待某个事件（如 I/O 完成、信号量），暂时无法运行 |
| **终止态 Terminated** | 进程执行完毕或被 kill，正在回收资源 |

### 状态转换图

```
                创建态
                  │
                  ▼
              就绪态 ◄──────────── 运行态 ─────────────► 终止态
                ▲                    │                       ▲
                │                    │                       │
                │   I/O完成/        │  等待I/O/             │
                │  事件发生         │  等待事件             │
                │                    ▼                       │
                └──────────────── 阻塞态 ────────────────────┘
                                              (可被信号终止)
```

**转换路径详解：**

1. **创建 → 就绪**：PCB 初始化完成，资源分配完毕，进程入就绪队列
2. **就绪 → 运行**：调度器选中该进程，分配 CPU
3. **运行 → 就绪**：时间片用完，或被更高优先级的进程抢占
4. **运行 → 阻塞**：正在等待 I/O 或事件（主动让出 CPU）
5. **阻塞 → 就绪**：I/O 完成或等待的事件发生（不会直接到运行态）
6. **运行 → 终止**：进程执行完毕或收到终止信号

### 面试题

**Q: 进程有哪些状态？状态如何转换？**

> **五种状态**：创建态 → 就绪态 → 运行态 → 阻塞态 → 终止态。
>
> **关键转换**：
> - 就绪→运行：调度器分配 CPU
> - 运行→就绪：时间片用完
> - 运行→阻塞：等待 I/O
> - 阻塞→就绪：I/O 完成（**不会直接到运行**）
> - 运行/阻塞→终止：执行完毕或被 kill
>
> **注意**：阻塞态不能直接到运行态，必须先经过就绪态。

### Linux 内核中的进程状态细分

上面是理论模型，Linux 内核实际实现时对"阻塞态"做了进一步区分，对排查问题很重要：

| Linux 状态 | 对应理论状态 | 含义 | `ps` 标识 |
|-----------|-------------|------|-----------|
| **TASK_RUNNING** | 就绪态 + 运行态 | 正在运行或在就绪队列排队 | **R** |
| **TASK_INTERRUPTIBLE** | 阻塞态 | 等待事件，**可被信号唤醒** | **S** |
| **TASK_UNINTERRUPTIBLE** | 阻塞态 | 等待事件，**不可被信号唤醒**（等待磁盘 I/O 常见）| **D** |
| **TASK_STOPPED** | — | 被 SIGSTOP 暂停 | **T** |
| **TASK_ZOMBIE** | 终止态 | 已终止但父进程未回收 | **Z** |

**为什么 D 状态很重要？**
- D 状态进程**计入 Load Average**（与 R 状态一起）
- 大量 D 状态进程 → Load Average 飙升但 CPU 不高 → **磁盘 I/O 瓶颈**
- D 状态进程无法被 kill（连 `kill -9` 都不行），必须等 I/O 完成
- 典型场景：NFS 挂载卡住、磁盘故障/慢盘、数据库大量随机 I/O

（Load Average 的详细分析见 [12.7 节](#127-load-average-详解-)）

## 2.3 线程基础 ★★★★★

### 什么是线程

线程（Thread）是进程内部的**执行流**，是 CPU **调度**的基本单位。一个进程可以包含多个线程，它们共享进程的地址空间和资源。

```
进程 = 资源分配单位（地址空间、文件、信号等）
线程 = CPU 调度单位（寄存器、栈、PC）
```

### 为什么需要线程

1. **提高并发性**：一个进程内多个任务可以"同时"执行（如浏览器同时渲染和下载）
2. **资源共享方便**：同进程的线程天然共享内存，不需要 IPC
3. **创建/切换开销小**：不需要重新分配地址空间
4. **充分利用多核**：多线程可以分别运行在不同 CPU 核心上
5. **响应性更好**：一个线程阻塞不会阻塞整个进程（如 GUI 程序）

### 面试题

**Q: 什么是线程？**

> 线程是**CPU 调度的基本单位**，是进程内部的一个执行流。一个进程可以包含多个线程，它们共享进程的代码段、数据段、堆、文件描述符等资源，但每个线程有自己的**栈**、**寄存器**和**程序计数器**。

**Q: 为什么线程切换比进程快？**

> 1. **不需要切换页表**：同一进程的线程共享地址空间，切换时不需要更换页表基址寄存器，TLB 不需要全刷新
> 2. **不需要切换内存映射**：线程共用同一个虚拟地址空间
> 3. **缓存更友好**：线程间共享的数据可能还在 CPU Cache 中
> 4. **上下文更小**：线程上下文只需要保存寄存器、PC、栈指针，而进程还要保存页表、文件描述符表等更多信息

**Q: 线程是不是越多越好？线程过多会有什么问题？**

> **不是**。线程过多带来的问题：
>
> 1. **内存开销**：每个线程有独立的栈（Linux 默认 8MB），1000 个线程 ≈ 8GB 栈空间。即使实际只用几十 KB，内核也必须预留
> 2. **调度开销**：线程数 >> CPU 核数时，大量时间花在上下文切换上，真正干活的时间占比下降。每个线程分到的时间片也更少
> 3. **缓存失效**：频繁切换线程导致 CPU Cache 反复失效（Cache Thrashing），每次切换回来缓存都是冷的
> 4. **锁竞争加剧**：线程越多，抢锁越激烈，更多线程阻塞等待 → 吞吐量反而下降
> 5. **内核资源耗尽**：每个线程消耗一个 PID、打开文件数、内核栈等，操作系统有上限
>
> **经验法则**：
> - **CPU 密集型**：线程数 ≈ CPU 核数（或核数 + 1）
> - **I/O 密集型**：线程数可以多一些（核数 × 2~4），但也不是无限
> - **Go 的优势**：Goroutine 初始栈 2KB 而非 8MB，可以安全创建数十万个

## 2.4 进程与线程区别 ★★★★★

| 维度 | 进程 | 线程 |
|------|------|------|
| **资源拥有** | 拥有独立的地址空间和其他资源 | 共享进程的地址空间和资源 |
| **CPU 调度** | 进程是资源分配单位 | 线程是 CPU 调度单位 |
| **开销** | 创建/切换开销大（需分配页表、复制资源） | 创建/切换开销小 |
| **通信方式** | IPC：管道、消息队列、共享内存、信号量、Socket | 直接读写共享内存（需同步） |
| **独立性** | 进程间相互独立，一个崩溃不影响其他 | 一个线程崩溃可能导致整个进程崩溃 |
| **地址空间** | 各自独立的虚拟地址空间 | 共享进程的地址空间 |
| **系统开销** | fork 需要复制页表（COW 优化） | clone 时共享地址空间 |

### 面试题

**Q: 进程和线程有什么区别？**

> 核心区别就一句话：**进程是资源分配的基本单位，线程是 CPU 调度的基本单位。**
>
> - **资源**：进程拥有独立的地址空间和资源；线程共享进程的地址空间和资源（每个线程有自己的栈和寄存器）
> - **开销**：进程创建/切换开销大（需切换页表）；线程开销小（共享地址空间）
> - **通信**：进程间需要 IPC（管道、共享内存等）；线程直接读写共享变量（但需加锁同步）
> - **健壮性**：进程间相互隔离，一个死掉不影响其他；一个线程 crash 很可能导致整个进程 crash

## 2.5 协程（Golang 重点） ★★★★★

### 什么是协程

协程（Coroutine）是一种**用户态**的轻量级线程，由用户程序自行调度，不需要内核参与。也叫"纤程（Fiber）"。

```
进程 > 线程 > 协程
         内核调度     用户调度
         切换在内核态   切换在用户态
         几 μs~几十 μs   几十 ns~几百 ns
```

### 用户态线程 vs 内核态线程

| 类型 | 调度者 | 切换开销 | 阻塞影响 |
|------|--------|---------|---------|
| **内核线程（1:1）** | 内核 | 大（需系统调用） | 一个阻塞 = 内核挂起该线程 |
| **用户线程（N:1）** | 用户态调度器 | 小 | 一个阻塞 = 所有线程阻塞 |
| **混合（M:N）** | 内核 + 用户 | 适中 | Goroutine 阻塞不会阻塞其他 G |

Go 的 GMP 模型正是**M:N 混合调度**：M 个 goroutine 映射到 N 个内核线程。

### Goroutine

Goroutine 是 Go 语言实现的协程：

- **栈动态增长**：初始只有 **2KB**（Go 1.4+，之前是 4KB），按需增长和收缩
- **用户态调度**：Go 运行时自行调度 G 到 M（内核线程）上执行
- **切换成本极低**：只需要保存/恢复少量寄存器（~几十 ns），不涉及系统调用
- **与系统线程对比**：线程栈固定 1MB~8MB，切换需要陷入内核（~1-10μs）

### 面试题

**Q: 协程和线程区别？**

> | 维度 | 协程 | 线程 |
> |------|------|------|
> | **调度者** | 用户程序/运行时 | 操作系统内核 |
> | **切换开销** | ~100ns 级 | ~1-10μs 级 |
> | **栈大小** | 少量 KB，动态伸缩（Goroutine 2KB） | 固定 1MB~8MB |
> | **数量** | 可以轻松创建数十万个 | 受限于内存（几千个） |
> | **阻塞影响** | 只阻塞当前协程（Go 中自动切换） | 阻塞整个线程 |
> | **调度时机** | 协作式/抢占式（Go 1.14+） | 抢占式 |
> | **地址空间** | 共享 | 共享 |

**Q: Goroutine 为什么轻量？**

> 1. **初始栈小**：Goroutine 初始栈仅 2KB，线程初始栈 1MB~8MB
> 2. **栈动态伸缩**：按需增长（每次翻倍），不需要时收缩
> 3. **用户态调度**：切换在用户态完成，不涉及系统调用，仅保存少量寄存器，~几十 ns
> 4. **G-P-M 模型**：M 个 Goroutine 复用 N 个内核线程，减少内核线程创建/切换开销
> 5. **Work Stealing**：空闲的 P 会从其他 P 的队列偷取 G，提高整体利用效率

## 2.6 线程上下文切换 ★★★★★

### 切换内容

线程上下文切换（Context Switch）发生时，操作系统需要保存当前线程的执行状态，并恢复下一个要运行的线程的状态。

**需要保存/恢复的内容：**

| 内容 | 说明 |
|------|------|
| **通用寄存器** | EAX/RAX, EBX/RBX, ECX/RCX, EDX/RDX 等 |
| **程序计数器 PC** | 当前执行到哪条指令（EIP/RIP） |
| **栈指针 SP** | 指向当前线程内核栈栈顶 |
| **标志寄存器** | EFLAGS/RFLAGS |
| **段寄存器** | CS, DS, ES, SS 等 |
| **浮点/向量寄存器** | 需要时惰性保存（FPU/SSE/AVX） |

如果是**进程切换**，还需要额外切换：
- **页表基址寄存器**（CR3）→ 切换地址空间
- 导致 **TLB 刷新**（除非使用 PCID/ASID 技术）

### 面试题

**Q: 上下文切换保存什么？**

> 保存当前线程/进程的执行上下文：通用寄存器、程序计数器 PC、栈指针 SP、标志寄存器、段寄存器。如果是进程切换还要切换页表基址寄存器 CR3（导致 TLB 刷新）。

**Q: 为什么上下文切换昂贵？**

> 1. **直接开销**：保存/恢复寄存器，切换内核栈，大约需要 1~10μs
> 2. **间接开销（Cache 污染）**：切换后 CPU Cache 中的数据大部分是"冷"的，新线程运行时大量 Cache Miss，严重影响性能。这是上下文切换**最昂贵**的部分
> 3. **TLB 刷新**：进程切换时需要刷新 TLB（快表），导致后续内存访问需要查页表（慢）
> 4. **流水线中断**：CPU 指令流水线被打断，需要重新填充
>
> **结论**：切换本身的操作不慢，慢的是**切换后带来的 Cache/TLB 失效**。

---

## 2.7 fork 与写时复制 COW ★★★★★

### fork 系统调用

```c
pid_t fork(void);
// 父进程中返回子进程 PID
// 子进程中返回 0
// 失败返回 -1
```

`fork()` 是 Linux 创建进程的唯一方式。调用一次，返回两次。

**fork 发生了什么？**

1. 为子进程分配新的 PID
2. **复制父进程的 task_struct**（PCB），然后修改子进程特有的字段（PID、PPID 等）
3. **复制父进程的页表**（不是复制全部内存！）——关键，下面详述
4. 复制父进程的文件描述符表（fd 表），但 fd 指向内核的同一个 file 对象
5. 子进程被创建后处于**就绪态**，等待调度

### 写时复制 COW（Copy On Write）

**问题**：如果 fork 直接复制父进程的全部内存，那创建进程的开销会非常大（一个 1GB 内存的进程 fork 需要复制 1GB）。

**解决方案——写时复制 COW**：

```
fork 时：
  - 只复制页表（~几 KB），不复制物理内存
  - 父、子进程的页表项都指向**同一块物理内存**
  - 将这些页都标记为**只读**（read-only）

写操作时：
  - 父/子进程尝试写共享页 → 触发页保护异常（写只读页）
  - 内核介入 → 真正的复制一页物理内存（4KB）
  - 修改页表映射 → 恢复写权限
  - 恢复进程执行
```

**效果**：
- fork 极快（只复制 PCB + 页表，不复制数据）
- 节省内存（父子共享大部分页面直到有一方写入）
- 很多页面可能永远不需要复制（如代码段 .text 只读不写）

### vfork 与 clone

| 调用 | 特点 |
|------|------|
| **fork** | 父子进程独立，COW 复制地址空间 |
| **vfork** | 子进程先运行，父进程阻塞；子进程不复制页表，共享父进程地址空间；子进程只能调用 exec/exit；已废弃 |
| **clone** | Linux 真正底层的进程/线程创建接口，可精确控制共享哪些资源（CLONE_VM、CLONE_FS、CLONE_FILES 等） |

Go 的 `go func()` 在底层调用 `clone` 创建轻量级进程（线程），而非 `fork`。

### 面试题

**Q: fork() 发生了什么？为什么 fork 很快？**

> fork 创建子进程时：
> 1. 分配新 PID，复制父进程的 PCB（task_struct）
> 2. 复制页表（不是物理内存页本身）
> 3. 将父子共享的物理页标记为**只读**（COW 机制）
> 4. 任一进程写入时触发缺页异常 → 内核真正复制该页（4KB）
>
> **fork 快的原因**：采用了写时复制（COW），只复制页表，不复制物理内存。即使父进程有 1GB 内存，fork 也只需复制页表（几 KB~几 MB）。

## 2.8 孤儿进程与僵尸进程 ★★★★★

### 孤儿进程（Orphan Process）

**定义**：父进程先于子进程终止，子进程变成"孤儿"。

**处理**：孤儿进程会被 **init 进程（PID=1）** 或 systemd "收养"，由 init 负责等待它们终止并回收资源。现代 Linux 可通过 `prctl(PR_SET_CHILD_SUBREAPER)` 让非 init 进程也能成为子进程收养者（Docker 容器中常用）。

```bash
# 模拟孤儿进程
bash -c 'sleep 100 &'  # 父进程 bash 退出后，sleep 被 init 收养
```

**孤儿进程没有危害**——系统会自动处理。

### 僵尸进程（Zombie Process）

**定义**：子进程已经终止，但父进程**没有调用 wait/waitpid** 来回收子进程的退出状态，导致子进程的 PCB（task_struct）无法被释放。

```
子进程死亡 → PCB 保留（等待父进程读取退出状态）
         → 父进程一直不 wait → 变成僵尸
```

**查看**：
```bash
ps aux | grep Z
# STAT 列为 Z → 僵尸进程
```

**危害**：
- 僵尸进程**不占内存**（已释放地址空间），但**占用 PID** 和 **PCB 槽位**
- PID 数量有限（`cat /proc/sys/kernel/pid_max`，默认 32768）
- 大量僵尸进程 → PID 耗尽 → 无法创建新进程

**解决方案**：
1. 父进程调用 `wait()` / `waitpid()` 回收子进程
2. 忽略 `SIGCHLD` 信号（`signal(SIGCHLD, SIG_IGN)`），子进程终止后自动回收
3. **杀死父进程** → 僵尸进程变成孤儿进程 → 被 init 收养并回收
4. 父进程中使用 `SA_NOCLDWAIT` 标志

```c
// 正确处理子进程退出的方式
signal(SIGCHLD, SIG_IGN);  // 最简单的方式
// 或者
void handler(int sig) {
    while (waitpid(-1, NULL, WNOHANG) > 0);
}
signal(SIGCHLD, handler);
```

### 面试题

**Q: 什么是僵尸进程？如何解决？**

> **僵尸进程**：子进程已终止，但父进程没有调用 wait() 回收退出状态，PCB 无法释放。僵尸进程不占内存但占用 PID，过多会导致 PID 耗尽。
>
> **解决**：
> 1. 父进程调用 `wait()` / `waitpid()` 回收
> 2. 忽略 `SIGCHLD` 信号让内核自动回收
> 3. 直接 kill 父进程（僵尸被 init 收养并回收）
> 4. Go 中不需要担心——runtime 自动回收

**Q: 孤儿进程和僵尸进程的区别？**

> | | 孤儿进程 | 僵尸进程 |
> |------|---------|---------|
> | **成因** | 父进程先死 | 父进程不 wait |
> | **状态** | 仍在运行 | 已终止 |
> | **危害** | 无（init 自动收养） | 有（占用 PID） |
> | **解决** | 无需解决 | 父进程 wait / kill 父进程 |

# 第三章 进程调度 ★★★★

## 3.1 调度基础

### 调度目标

| 目标 | 含义 |
|------|------|
| **CPU 利用率** | 让 CPU 尽可能忙（CPU-bound 场景） |
| **吞吐量** | 单位时间完成的进程数量 |
| **周转时间** | 从进程提交到完成的总时间 |
| **等待时间** | 进程在就绪队列中等待的时间 |
| **响应时间** | 从提交到第一次响应的时间（交互式场景） |
| **公平性** | 每个进程都能获得合理的 CPU 时间 |

### 吞吐量

单位时间内系统完成的作业数。与进程的**平均执行时间**成反比。

### 响应时间

从用户输入到系统首次响应的时间间隔。对**交互式系统**（桌面、Web 服务器）至关重要。

### 周转时间

周转时间 = 完成时间 - 到达时间。包含等待时间 + 执行时间 + I/O 时间。

---

## 3.2 常见调度算法 ★★★★★

### FCFS（First Come First Serve，先来先服务）

- **非抢占式**：按进程到达顺序依次分配 CPU
- **优点**：简单、公平
- **缺点**：长进程会阻塞后面所有短进程（**护航效应/Convoy Effect**）
- 平均等待时间通常较大

### SJF（Shortest Job First，短作业优先）

- 优先执行**预计执行时间最短**的进程
- **非抢占式 SJF**：一旦开始执行就不可中断
- **抢占式 SJF（SRTF，最短剩余时间优先）**：新来更短的进程会抢占 CPU
- **优点**：平均等待时间最小（理论上最优）
- **缺点**：需要预知进程的执行时间（几乎不可能），长作业可能**饿死**

### RR（Round Robin，时间片轮转）

- 就绪队列按 FCFS 排列，每个进程被分配固定的**时间片**（quantum，通常 10~100ms）
- 时间片用完未执行完 → 放入队尾，切换到下一个进程
- **优点**：公平，响应时间可控，适合交互式系统
- **缺点**：时间片太小 → 上下文切换过多；时间片太大 → 退化为 FCFS
- **经验值**：时间片应远大于上下文切换时间（~10μs），通常选**10~50ms**

### 优先级调度

- 为每个进程分配一个**优先级**，CPU 分配给最高优先级的就绪进程
- 可以是**抢占式**（高优先级到来直接抢占）或**非抢占式**
- **静态优先级**：创建时确定，不变
- **动态优先级**：运行过程中调整（如 I/O 密集型进程优先级提升）
- **缺点**：低优先级进程可能**饿死** → 使用**老化（Aging）**解决

### 多级反馈队列（MLFQ） ★★★★★

**现代操作系统实际采用的调度算法**（Linux CFS 也是演化自这个思想）。

**设计规则：**

1. 系统维护**多个优先级队列**（Q0 优先级最高，Qn 最低）
2. 新进程进入**最高优先级队列**
3. 高优先级队列的时间片**小**（如 Q0=8ms），优先级越低时间片**越大**（如 Q1=16ms）
4. 一个进程用完了当前队列的时间片还没执行完 → **降级**到下一个队列
5. 当有更高优先级队列的进程就绪时 → **抢占**当前进程
6. 定期将所有进程提升到最高优先级 → 防止饥饿（**优先级提升/Aging**）

**工作原理：**

```
  新进程
    │
    ▼
 ┌──────┐  时间片用完   ┌──────┐  时间片用完   ┌──────┐
 │ Q0   │ ────────────→ │ Q1   │ ────────────→ │ Q2   │ ...
 │ 8ms  │               │ 16ms │               │ 32ms │
 └──────┘               └──────┘               └──────┘
    ▲                                              │
    │        I/O 请求                               │
    │        时间片用完                              │
    └──────────────────────────────────────────────┘
```

**为什么 MLFQ 好？**
- **I/O 密集型进程**：频繁 I/O，没等用完时间片就阻塞 → 留在高优先级队列 → 获得快速响应
- **CPU 密集型进程**：持续用完时间片 → 逐渐降到低优先级 → 不阻塞交互进程
- 兼顾**交互响应快**和**吞吐量大**

### 面试题

**Q: Linux 使用什么调度算法？**

> Linux 2.6+ 使用 **CFS（Completely Fair Scheduler，完全公平调度器）**。
>
> **核心思想**：为每个进程维护一个**虚拟运行时间（vruntime）**，vruntime 越小的进程越需要 CPU，调度器始终选择 vruntime 最小的进程运行。
>
> - vruntime = 实际运行时间 × (1024 / 进程权重)，nice 值是权重依据
> - 使用**红黑树**组织就绪进程，按 vruntime 排序
> - 选取最左边的节点（vruntime 最小）运行
> - CFS 没有固定时间片，动态计算**目标延迟**和**最小粒度**
>
> 这不是 MLFQ，但思想类似：I/O 密集型（vruntime 增长慢）自然获得更多调度机会。

**Q: 多级反馈队列如何工作？**

> MLFQ 维护多个优先级队列，规则如下：
> 1. 新进程进入最高优先级队列
> 2. 高优先级队列的时间片短，低优先级的时间片长
> 3. 用完当前队列时间片 → 降级
> 4. I/O 阻塞 → 留在当前级别（获得快速响应）
> 5. 定期优先级提升 → 防止饥饿
>
> 核心思想：**根据进程行为自动调整优先级**——I/O 密集型留高优先，CPU 密集型逐渐降级。

# 第四章 进程通信 IPC ★★★★★

## 4.1 为什么需要进程通信

进程有独立的地址空间，一个进程不能直接访问另一个进程的内存。为了让不同进程协同工作，就需要**进程间通信（Inter-Process Communication, IPC）**机制。

---

## 4.2 管道 Pipe ★★★★★

### 匿名管道（Anonymous Pipe）

```c
int pipe(int fd[2]);  // fd[0] 读端，fd[1] 写端
```

**特点：**
- **半双工**：数据只能单向流动
- **只能在父子进程或兄弟进程间使用**（通过 fork 传递 fd）
- 数据在内存缓冲区中，按**字节流**传输
- **阻塞式**读写：管道空时 read 阻塞，管道满时 write 阻塞
- 进程终止 → 管道自动销毁

### 命名管道（FIFO）

```bash
mkfifo mypipe    # 创建一个命名管道
echo "hello" > mypipe   # 写端
cat < mypipe            # 读端
```

**特点：**
- 有一个**文件名**存在于文件系统中
- **任意进程**都可以通过文件名打开该管道
- 支持无亲缘关系进程间通信
- 仍然是半双工，需要双向通信则创建两个 FIFO

### 面试题

**Q: 管道有什么特点？**

> 1. **半双工**：数据单向流动，双向需要两根管道
> 2. **面向字节流**：没有消息边界
> 3. **阻塞式**：空管道 read 阻塞，满管道 write 阻塞
> 4. **匿名管道**：只能在亲缘进程间使用（fork 传递 fd）
> 5. **命名管道 FIFO**：通过文件名标识，任意进程可访问
> 6. **生命周期随进程**：最后一个引用关闭后管道销毁
> 7. **存在于内存**：数据在 kernel 缓冲区，不落盘
> 8. **容量有限**：Linux 管道容量默认 64KB（可通过 fcntl 修改）

---

## 4.3 消息队列 ★★★★

### 工作原理

消息队列是**存放在内核中的消息链表**，进程可以向队列发送消息，也可以从队列接收消息。

```c
int msgget(key_t key, int msgflg);          // 创建队列
int msgsnd(int msqid, const void *msgp, size_t msgsz, int msgflg);  // 发送
int msgrcv(int msqid, void *msgp, size_t msgsz, long msgtype, int msgflg);  // 接收
```

### 优缺点

| 优点 | 缺点 |
|------|------|
| 有**消息边界**（不像管道是字节流） | 每条消息有长度上限 |
| 支持按**消息类型**分类接收 | 消息队列有总容量限制 |
| 异步通信，发送方不需要等待接收方 | 存在内核态↔用户态数据拷贝 |
| 独立于进程生命周期 | 不及时取走 → 队列满 → 发送方阻塞 |

---

## 4.4 共享内存 ★★★★★

### 工作原理

共享内存是**最快的 IPC 方式**。不同进程通过各自的页表将一段物理内存映射到各自的虚拟地址空间中。此后，进程对该内存的读写就像访问自己的内存一样，**不需要内核参与**。

```c
int shmget(key_t key, size_t size, int shmflg);  // 创建共享内存
void *shmat(int shmid, const void *shmaddr, int shmflg);  // 映射到地址空间
int shmdt(const void *shmaddr);                  // 解除映射
```

### 优缺点

| 优点 | 缺点 |
|------|------|
| **不需要数据拷贝** | 需要**同步机制**（信号量/互斥锁） |
| 访问速度最快（~内存速度） | 多进程同时写 → 数据不一致 |
| 适合大数据量、高频通信 | 使用不当容易造成死锁 |

### 面试题

**Q: 为什么共享内存最快？**

> 其他 IPC 方式（管道、消息队列、Socket）都需要**两次拷贝**：
> 用户空间 → 内核空间 → 另一个进程的用户空间。
>
> 共享内存只需要：**建立映射时操作页表 + MMU 做地址翻译**，后续数据读写直接访问物理内存，**不需要任何内核参与，零拷贝**。
>
> 但共享内存需要额外使用信号量/互斥锁做同步。

---

## 4.5 信号量 ★★★★★

### P 操作（wait / down / lock）

```c
P(S) {
    while (S <= 0) ;  // 忙等（实际上会阻塞）
    S--;
}
```

- 尝试获取资源
- S > 0：获取成功，S--
- S <= 0：进程阻塞（放入等待队列）

### V 操作（signal / up / unlock）

```c
V(S) {
    S++;
    // 如果有进程在等待，唤醒一个
}
```

- 释放资源
- S++，并唤醒一个等待的进程

### 信号量分类

| 类型 | 说明 |
|------|------|
| **互斥信号量（二元信号量）** | 取值 0/1，用于互斥访问临界区 |
| **计数信号量** | 取值 ≥0，用于控制资源访问数量 |

### 面试题

**Q: 信号量是什么？**

> 信号量是一个**整型变量**，用于多进程/多线程之间的同步与互斥。它有两个原子操作：
> - **P 操作（wait）**：尝试获取资源，S > 0 时 S--，否则阻塞
> - **V 操作（signal）**：释放资源，S++，唤醒等待进程
>
> 信号量初始化为 1 → **互斥锁**（Mutual Exclusion）
> 信号量初始化为 0 → **同步**（一个进程等待另一个事件）
> 信号量初始化为 N → **资源计数器**（最多 N 个进程同时访问）

---

## 4.6 Socket 通信 ★★★★★

### 本地 Socket（Unix Domain Socket）

```go
// Go 示例：本地 Socket
conn, _ := net.Dial("unix", "/tmp/app.sock")
```

- 用于**同一台机器**上的进程间通信
- 路径名称标识，不走网络协议栈
- **不需要经过网卡**，比 TCP Socket 更快
- 支持字节流（SOCK_STREAM）和数据报（SOCK_DGRAM）

### 网络 Socket

- 不同主机上的进程通过 TCP/UDP Socket 通信
- 使用 IP + 端口号标识

### 面试题

**Q: Socket 为什么属于 IPC？**

> Socket 可以实现**同一台机器上不同进程的通信**（Unix Domain Socket），也能实现**不同机器之间的进程通信**（TCP/UDP Socket）。IPC（进程间通信）的定义并不限制进程是否在同一台机器上，只要能实现数据交换，就是 IPC。Socket 正是这样的一种通用通信方式。

## 4.7 信号 Signal ★★★★

### 常见信号

| 信号 | 编号 | 含义 | 默认行为 |
|------|------|------|---------|
| SIGINT | 2 | Ctrl+C 中断 | 终止进程 |
| SIGQUIT | 3 | Ctrl+\ 退出 | 终止 + core dump |
| SIGKILL | 9 | 强制终止（**不可捕获/忽略**） | 终止进程 |
| SIGTERM | 15 | 终止信号（默认） | 终止进程 |
| SIGSTOP | 19 | 暂停进程（**不可捕获/忽略**） | 暂停进程 |
| SIGCONT | 18 | 继续运行暂停的进程 | 继续/忽略 |
| SIGCHLD | 17 | 子进程状态变化 | 忽略 |
| SIGPIPE | 13 | 向已关闭的管道写数据 | 终止进程 |
| SIGSEGV | 11 | 内存访问越界（段错误） | 终止 + core dump |

### 面试题

**Q: kill -9 和 kill -15 的区别？**

> - `kill -15`（SIGTERM）：**优雅终止**。程序可以**捕获**该信号，执行清理操作（关闭连接、保存数据、删除临时文件等），然后主动退出。可以被阻塞/忽略。
> - `kill -9`（SIGKILL）：**强制终止**。内核直接杀死进程，**程序无法捕获也不能忽略**。进程没有任何清理机会，可能造成数据丢失、连接未断开等问题。
>
> **实际使用**：先 `kill -15`，等待一段时间；进程不响应再用 `kill -9`。

## 4.8 线程间通信方式 ★★★★

同一进程内的线程**共享地址空间**，因此线程间通信本质上是**直接访问共享数据**。但需要同步机制保证安全。

### 线程间通信方式汇总

| 方式 | 说明 | 典型场景 |
|------|------|---------|
| **共享全局变量** | 直接读写进程堆中的全局变量（需加锁） | 最简单的数据共享 |
| **互斥锁 Mutex** | 保证同一时刻只有一个线程访问临界区 | 修改共享状态 |
| **读写锁 RWMutex** | 读共享、写独占 | 缓存读写、配置读取 |
| **条件变量 Cond** | 一个线程等待条件，另一个线程通知 | 生产者-消费者、任务队列 |
| **信号量 Semaphore** | 控制同时访问资源的线程数 | 连接池限流、并发度控制 |
| **原子操作 atomic** | 无锁地更新单个变量 | 计数器、状态标志 |
| **Channel**（Go） | Go 语言原生协程通信方式 | Goroutine 间传递数据 |
| **线程局部存储 TLS** | 每个线程拥有变量的私有副本 | 线程上下文、事务 ID |

### 线程通信 vs 进程通信

| 维度 | 线程间通信 | 进程间通信 |
|------|-----------|-----------|
| **根本差异** | 共享地址空间 | 地址空间隔离 |
| **主要方式** | 直接读写共享内存 + 同步 | 管道、消息队列、共享内存、Socket 等 |
| **速度** | 极快（内存读写） | 较慢（需内核参与，除共享内存外） |
| **同步需求** | 必须（锁/原子操作） | 部分需要（共享内存需额外同步） |
| **复杂度** | 简单（直接访问） | 复杂（需 IPC 机制） |

### Go Channel 内部实现 ★★★★★

Channel 是 Go 中 goroutine 间通信的核心机制。其底层结构 `hchan` 包含：

```go
type hchan struct {
    qcount   uint            // 当前缓冲区中元素个数
    dataqsiz uint            // 缓冲区总大小（容量）
    buf      unsafe.Pointer  // 环形缓冲区指针
    elemsize uint16          // 每个元素大小
    closed   uint32          // 是否已关闭
    sendx    uint            // 发送索引（buf 中的写入位置）
    recvx    uint            // 接收索引（buf 中的读取位置）
    recvq    waitq           // 等待接收的 goroutine 队列
    sendq    waitq           // 等待发送的 goroutine 队列
    lock     mutex           // 保护 hchan 的互斥锁
}
```

**核心机制**：

1. **环形缓冲区（ring buffer）**：有缓冲 channel 的数据存储，sendx/recvx 指针循环移动
2. **等待队列**：无缓冲或满/空时，goroutine 挂起到 sendq/recvq
3. **直接拷贝**：channel 传递数据总是**值拷贝**（不是共享指针），天然避免数据竞争
4. **happens-before 保证**：`ch <- data` happens-before `<-ch` 对应的接收完成

```
无缓冲 channel:           有缓冲 channel:
send G ──► 阻塞等待         send G ──► 拷贝到 buf → 立即返回
  │          (配对接收)       │
  ▼                           │          (buf已满 → 加入sendq阻塞)
recv G ──► 阻塞等待           ▼
  │                           recv G ──► 从 buf 取数据
  ▼                                        (buf空 → 加入recvq阻塞)
数据直接从 send G 栈
拷贝到 recv G 栈
（无中间缓冲）
```

**关闭语义**：
- `close(ch)` 后，接收方读到零值 + `ok=false`
- 向已关闭 channel 发送 → **panic**
- 重复关闭 → **panic**
- 从已关闭的空 channel 接收 → 零值（不阻塞）

**select 语句**：运行时对每个 case 加锁检查是否就绪，如果都不就绪则阻塞，任意一个就绪时执行对应分支。多 case 同时就绪时**伪随机**选择（防止饿死）。

> **Go 哲学**：不要通过共享内存通信，而要通过通信共享内存。Channel 的设计让数据所有权在 goroutine 间转移，而非共享。

# 第五章 线程同步与互斥 ★★★★★

## 5.1 竞态条件（Race Condition）

当多个线程**并发访问共享数据**，并且最终结果依赖于**线程执行的具体顺序**时，就产生了竞态条件。

```go
// 竞态条件示例
var counter int

func increment() {
    counter++  // 不是原子操作：读-改-写 三步
}

// 两个 goroutine 同时执行，counter 可能少于预期
```

**避免方法**：互斥锁、原子操作、无锁数据结构、不可变数据。

### 线程安全 vs 可重入（Thread-Safe vs Reentrant）

面试中经常混淆的两个概念：

| 维度 | 线程安全（Thread-Safe） | 可重入（Reentrant） |
|------|------------------------|---------------------|
| **定义** | 多线程并发调用不会出现数据竞争 | 一个函数在被中断后可以安全地再次被调用 |
| **保护方式** | 使用互斥锁/原子操作等同步原语 | 不依赖全局/静态数据，或使用原子操作 |
| **能否用锁实现** | ✅ 可以 | ❌ 不能用锁（会导致死锁） |
| **典型场景** | 普通多线程编程 | **信号处理函数**、递归调用 |
| **关系** | 可重入函数**一定是**线程安全的 | 线程安全函数**不一定**可重入 |

```
线程安全但不一定可重入的例子：

void thread_safe_func() {
    static Mutex mu;
    mu.Lock();          // ← 如果信号处理器在同一线程中再次调用此函数
    // ...                 //    → 死锁！因为锁已被持有
    mu.Unlock();
}

// 如果此函数被用做 SIGINT 信号处理函数
// 且主程序也在调用此函数 → 死锁
```

**面试要点**：`malloc` 是线程安全的（内部有锁），但**不可重入**——信号处理函数中调用 `malloc` 可能死锁。`errno` 在多线程中通过 TLS（线程局部存储）实现，每个线程有独立的 `errno`。

## 5.2 临界区（Critical Section）

临界区是指**访问共享资源的那段代码**。临界区设计原则：

1. **互斥进入**：同一时刻最多只有一个线程在临界区内
2. **有空让进**：临界区空闲时应让请求者立即进入
3. **有限等待**：线程申请进入后必须在有限时间内获准

---

## 5.3 互斥锁 Mutex ★★★★★

### 加锁 / 解锁

```go
var mu sync.Mutex
mu.Lock()    // 加锁，如果已被锁则阻塞等待
// ... 临界区 ...
mu.Unlock()  // 解锁
```

**原理**：基于原子操作（如 CAS）+ 内核的 futex 机制。无竞争时在用户态完成（~几十 ns）；有竞争时通过 futex 陷入内核阻塞（~几 μs）。

### 面试题

**Q: 什么是互斥锁？**

> 互斥锁（Mutex，Mutual Exclusion）是保证**同一时刻只有一个线程访问临界区**的同步机制。线程进入临界区前 `Lock`（锁被占用则阻塞），离开时 `Unlock`。它解决了竞态条件，保证了共享数据的一致性。
>
> **注意**：Mutex 保护的是**临界区代码**，不是具体的数据。

## 5.4 自旋锁 SpinLock ★★★★★

### 工作原理

自旋锁在获取锁失败时，**不会让出 CPU**，而是在一个循环中不断检查锁是否可用（"自旋"）。

```go
// 伪代码
type SpinLock struct {
    locked int32  // 0=未锁，1=已锁
}

func (s *SpinLock) Lock() {
    for !atomic.CompareAndSwapInt32(&s.locked, 0, 1) {
        // 自旋等待
        runtime.Gosched() // 或者 pause 指令
    }
}
```

### 使用场景

| 自旋锁适用 | 互斥锁适用 |
|-----------|-----------|
| 临界区**极短**（几条指令） | 临界区较长 |
| 锁竞争**不激烈** | 锁竞争激烈 |
| 不能睡眠的场景（中断上下文） | 普通用户态代码 |

### 面试题

**Q: 自旋锁和互斥锁的区别？**

> | 维度 | 自旋锁 | 互斥锁 |
> |------|--------|--------|
> | **等待方式** | 忙等（占用 CPU 循环检查） | 阻塞（让出 CPU，进入等待队列） |
> | **CPU 消耗** | 等待期间 100% 占用 CPU | 等待期间不占 CPU |
> | **上下文切换** | 无 | 有（阻塞/唤醒需要系统调用） |
> | **适用场景** | 临界区极短（< 上下文切换时间） | 临界区较长 |
> | **使用环境** | 中断上下文也可用 | 只能在进程上下文 |
> | **实现** | 纯用户态 CAS 循环 | 基于 futex，有竞争时陷入内核 |
>
> **一句话**：临界区很短用自旋锁，临界区长用互斥锁。

## 5.5 读写锁 RWLock ★★★★

### 读共享、写独占

```go
var rw sync.RWMutex

// 读锁：允许多个 goroutine 同时持有
rw.RLock()
// ... 读操作 ...
rw.RUnlock()

// 写锁：独占，与读锁/写锁互斥
rw.Lock()
// ... 写操作 ...
rw.Unlock()
```

**规则：**

| | 读锁 | 写锁 |
|------|------|------|
| **读锁** | ✅ 兼容 | ❌ 互斥 |
| **写锁** | ❌ 互斥 | ❌ 互斥 |

**读写锁适用于读多写少的场景**，可以显著提高并发度。

### wait / signal / broadcast

cond := sync.NewCond(&mu)

## 5.7 CAS ★★★★★

### Compare And Swap

CAS 是一条**CPU 原子指令**（cmpxchg），含义是：

```
CAS(addr, old, new):
    if *addr == old:
        *addr = new
        return true
    else:
        return false
```

这个操作是**原子的**（硬件保证），是构建无锁数据结构和锁的基础。

### ABA 问题

**场景**：
1. 线程 1 读 A 值为 X
2. 线程 2 将 A 从 X 改为 Y，再改回 X
3. 线程 1 CAS(A, X, Z) → **成功**（但 A 实际上已经被修改过！）

**解决方案**：使用**版本号/标记**（如 Go 的 `atomic.Value` 或指针+计数器）：

```go
// 解决 ABA：使用 AtomicStampedReference 或双字 CAS
// Double-Word CAS: 同时比较和交换指针 + 版本号
```

### 面试题

**Q: CAS 原理是什么？**

> CAS（Compare And Swap）是**CPU 提供的原子指令**，操作是：
> 比较内存地址的当前值是否等于旧值，相等则更新为新值，否则不操作。
> 整个过程是**原子的**，不会被其他线程打断。
>
> 用途：实现原子操作、自旋锁、无锁数据结构（无锁队列、无锁栈）。
>
> **ABA 问题**：值从 A→B→A，CAS 检测不到中间变化。
> **解决**：加版本号 → 变成比较"值 + 版本号"的双字 CAS。

## 5.8 乐观锁与悲观锁 ★★★★★

### 面试题

**Q: 乐观锁和悲观锁的区别？**

> | 维度 | 悲观锁 | 乐观锁 |
> |------|--------|--------|
> | **思想** | "一定会有人改" | "大概率没人改" |
> | **实现** | 先加锁再操作 | 先操作，提交时检查冲突 |
> | **具体** | Mutex, synchronized, SELECT FOR UPDATE | CAS, 版本号, 时间戳 |
> | **冲突少时** | 加锁开销大 | 高效（无锁） |
> | **冲突多时** | 适合 | 大量重试，性能差 |
> | **实现复杂度** | 低 | 高（需要重试逻辑） |
> | **典型场景** | 对一条记录加锁修改（电商库存扣减） | 并发读多写少（配置更新） |
>
> **DB 中的实现**：
> - 乐观锁：`UPDATE t SET v=v+1, version=version+1 WHERE id=1 AND version=5`
> - 悲观锁：`SELECT ... FOR UPDATE` 然后更新

## 5.9 伪共享 False Sharing ★★★★

### 什么是伪共享

伪共享（False Sharing）是**多核 CPU 缓存一致性问题**导致的一种性能退化现象：多个线程修改**不同的变量**，但这些变量恰好位于**同一个 CPU Cache Line** 中，导致缓存行在 CPU 核心之间反复失效和同步，像"假共享"一样产生竞争。

```
CPU 缓存行（Cache Line）大小通常为 64 字节

┌────────────────── 64 bytes ──────────────────┐
│ var A (线程1写) │ var B (线程2写) │ ...        │
└──────────────────────────────────────────────┘
        ↑                  ↑
      CPU0               CPU1
     修改A               修改B
     → 缓存行标记为脏     → 缓存行失效
     → CPU1 需重新加载   → CPU0 需重新加载
```

虽然线程 1 和线程 2 访问的是**不同的变量**，但因为它们落在同一个 Cache Line 中，CPU 的**缓存一致性协议（MESI）**会导致该 Cache Line 在核间反复跳动，造成严重的性能衰退。

### 解决方法

**1. 缓存行填充（Cache Line Padding）**

```go
// Go 中：在热变量之间填充无用字节，确保它们不在同一 Cache Line
type PaddedCounter struct {
    value int64
    _     [56]byte  // padding 到 64 字节
}

// 标准库中的做法
type Pad struct {
    _ [56]byte
}

type Counter struct {
    _     Pad
    value int64
    _     Pad  // 避免相邻变量影响
}
```

**2. 使用局部变量**

每个线程将共享变量**读入自己的局部变量**中操作，最后再合并。

**3. 结构体字段重排**

将对不同线程可能同时访问的字段分散到不同的 Cache Line。

### 面试题

**Q: 什么是伪共享？如何解决？**

> 伪共享是多核 CPU 中，不同线程修改**不同变量**但它们在**同一 Cache Line（64字节）** 中，导致缓存行在核间反复失效传输，产生性能退化。
>
> **解决**：填充（Padding）使变量独占一个 Cache Line（Go 中补 56 字节）；或改为局部变量避免共享。
>
> **常见场景**：高并发计数器、无锁队列的 head/tail 指针、Go channel 的内部 ring buffer。

## 5.10 Futex 机制 ★★★★

### 什么是 Futex

**Futex（Fast Userspace muTEX）** 是 Linux 提供的一种**混合锁机制**，是现代互斥锁（pthread_mutex、Go sync.Mutex 等）的底层基础。

```
无竞争时：纯用户态操作（CAS）        → 极快，不涉及内核
有竞争时：通过 futex 系统调用阻塞    → 内核将线程放入等待队列
```

**工作流程：**

```
Lock():
  1. CAS(锁变量, 0, 1)  →  成功 → 获得锁（用户态，~几十ns）
  2. CAS 失败 →  调用 futex(FUTEX_WAIT) 陷入内核
  3. 内核将线程加入等待队列 → 挂起

Unlock():
  1. CAS(锁变量, 1, 0)  →  成功
  2. 如果有等待者 →  调用 futex(FUTEX_WAKE) 唤醒一个等待线程
```

### 为什么 Futex 好

| 方案 | 无竞争开销 | 有竞争行为 |
|------|-----------|-----------|
| **纯自旋锁** | 低 | CPU 空转浪费 |
| **纯内核锁** | 高（每次都要系统调用） | 优雅阻塞 |
| **Futex** | 低（纯用户态 CAS） | 优雅阻塞（只竞争时进内核） |

**Futex 的精髓**：**将"用不着的内核开销"推迟到真正需要的时候**。

### Futex 在 Go 中的应用

Go 的 `sync.Mutex` 在 Linux 上的实现链路：

```
sync.Mutex.Lock()
  → runtime.lock2()        // Go runtime 内部的锁
    → atomic.Cas()         // 先快速路径 CAS
    → runtime.semasleep()  // CAS 失败 → 信号量等待
      → futex(WAIT)        // Linux futex
```

### 面试题

**Q: Futex 是什么？为什么用 Futex？**

> Futex（Fast Userspace muTEX）是 Linux 提供的混合锁机制。**无竞争时纯用户态 CAS（几十 ns），有竞争时才通过 futex 系统调用陷入内核阻塞（几 μs）**。
>
> 它结合了自旋锁（快）和互斥锁（优雅阻塞）的优点，是现代互斥锁的底层基础。Go sync.Mutex、pthread_mutex、Java ReentrantLock 在 Linux 上的实现都依赖 futex。

## 5.11 管程 Monitor ★★★★

### 什么是管程

管程（Monitor）是一种**高级同步机制**，将共享数据、操作共享数据的方法、互斥锁和条件变量**封装在一起**，简化并发编程。

```
管程 = 共享数据 + 互斥访问 + 条件变量（等待/通知）

管程保证：同一时刻最多一个线程在管程内部执行
```

### Java synchronized 本质就是管程

```java
// Java 中，每个对象关联一个 Monitor
synchronized (obj) {
    // 进入管程：获取 obj 的 Monitor 锁
    // 临界区代码
    obj.wait();   // 释放锁 + 进入等待队列
    obj.notify(); // 唤醒一个等待线程
    // 离开管程：释放 Monitor 锁
}
```

### Mesa 语义 vs Hoare 语义

管程中 `notify` 之后的语义有两种设计，这直接影响 `wait()` 应该用 `while` 还是 `if`：

| | Hoare 语义 | Mesa 语义 |
|------|-----------|----------|
| **notify 后** | 立即切换到被唤醒线程 | 继续执行当前线程，被唤醒线程进入就绪队列等待 |
| **被唤醒时条件** | 保证条件成立 | 条件可能已被其他线程再次改变 |
| **wait 写法** | `if (!condition) wait()` | `while (!condition) wait()` |
| **采用语言** | 理论研究为主 | Java, Go, pthread |

**为什么 Java wait() 必须用 while 而不是 if？**

```java
// ❌ 错误写法
synchronized (lock) {
    if (queue.isEmpty()) {
        lock.wait();  // 被唤醒后直接执行，条件可能已失效
    }
    queue.remove();   // 可能抛异常！
}

// ✅ 正确写法（Mesa 语义下）
synchronized (lock) {
    while (queue.isEmpty()) {
        lock.wait();  // 被唤醒后重新检查条件
    }
    queue.remove();   // 确认条件成立
}
```

Go 的 `sync.Cond.Wait()` 同样使用 Mesa 语义，**必须放在 for 循环中**。

### 面试题

**Q: 为什么 wait() 必须用 while 而不是 if？**

> 因为主流语言（Java、Go、pthread）使用 **Mesa 语义**：notify 后不立即切换执行，被唤醒线程需排队重新获取锁。在等待锁的间隙，条件可能已被其他线程改变。因此被唤醒后必须用 `while` **重新检查条件**，防止"虚假唤醒"（spurious wakeup）。

## 5.12 CPU 缓存一致性 MESI ★★★★

### 多核 CPU 的缓存问题

```
Core0 修改了变量 x = 1
  → x 在 Core0 的 L1 Cache 中是 1
  → x 在 Core1 的 L1 Cache 中还是旧值 0
  → Core1 读取 x → 读到 0！数据不一致！
```

### MESI 协议

MESI 是 CPU 缓存一致性协议，每个 **Cache Line（64B）** 处于四种状态之一：

| 状态 | 含义 | 说明 |
|------|------|------|
| **M**odified（已修改） | 只在本核缓存中，已被修改 | 与内存不一致，**写回前其他核不可读** |
| **E**xclusive（独占） | 只在本核缓存中，与内存一致 | 可以直接修改而不通知其他核 |
| **S**hared（共享） | 可能在多个核的缓存中 | 与内存一致，**要写必须先使其他核失效** |
| **I**nvalid（无效） | 该缓存行已失效 | 读取时必须从其他核或内存重新加载 |

**状态转换示例：**

```
Core0 读取 x：
  → x 不在 Core0 Cache → 从内存加载 → 状态 E（独占本核）

Core1 也读取 x：
  → Core0 的 x: E → S（共享）
  → Core1 加载 x → 状态 S

Core0 写入 x：
  → 必须先使 Core1 的 x 失效 → RFO(Request For Ownership) 消息
  → Core1 的 x: S → I（失效）
  → Core0 的 x: S → M（独占修改）
  → 写入完成
```

### 面试题

**Q: MESI 协议是什么？**

> MESI 是 CPU 缓存一致性协议，保证多核之间缓存数据一致。每个 Cache Line 有四种状态：Modified（独占已改）、Exclusive（独占干净）、Shared（共享干净）、Invalid（无效）。
>
> 写共享数据时，必须先通过 RFO 消息使其他核的副本失效（→ I），再修改自己的副本（→ M）。这解释了伪共享的性能问题：不同线程写同一 Cache Line 的不同变量，会导致该行在两个核之间反复 M→I→M→I 跳变。

## 5.13 RCU（Read-Copy-Update）★★★★

### 什么是 RCU

RCU 是 Linux 内核最重要的**无锁同步机制**之一，允许**多个读者和一个写者**并发操作而无需加锁。读者永远不阻塞，写者通过"复制-修改-替换指针"的方式更新数据。

```
传统读写锁                        RCU
┌──────────┐                    ┌──────────┐
│ Reader 1 │ ── RLock ──►      │ Reader 1 │ ── 直接读（无锁）──►
│ Reader 2 │ ── RLock ──►      │ Reader 2 │ ── 直接读（无锁）──►
│ Writer   │ ── Lock ───►      │ Writer   │ ── 复制新版本 ──►
│          │   (阻塞等读者)      │          │    更新指针
│          │                    │          │    旧版本延迟释放
└──────────┘                    └──────────┘
   读者阻塞写者                    读者永不阻塞！
```

### 核心操作

| 操作 | 说明 |
|------|------|
| **rcu_read_lock()** | 进入 RCU 读临界区（实际上只是禁止抢占，不获取任何锁） |
| **rcu_read_unlock()** | 退出 RCU 读临界区 |
| **rcu_assign_pointer()** | 写者更新指针（使用内存屏障保证顺序） |
| **synchronize_rcu()** | 等待所有读者退出临界区（阻塞写者） |
| **call_rcu()** | 注册回调，所有读者退出后自动释放旧数据（不阻塞写者） |

### 宽限期（Grace Period）

```
CPU0: 读者A ─────────┐       读者B ─────┐
                      │                  │
CPU1:        读者C ───┘  读者D ───┐      │
                                  │      │
时间: ─────────────────────────────────────→
       │         宽限期(Grace Period)       │
       │←─────────────────────────────────→│
        ↑                                  ↑
   写者开始更新                         所有旧读者已退出
   旧数据还不能释放                       安全释放旧数据
```

宽限期定义：从写者更新指针那一刻开始，到所有在此之前进入 RCU 读临界区的读者全部退出的时间段。宽限期结束后，旧版本数据安全释放。

### 适用场景

| 场景 | 说明 |
|------|------|
| **Linux 内核** | 网络路由表、文件系统 dentry cache、模块卸载 |
| **数据结构** | 链表、hash 表、radix tree 的并发访问 |
| **特点** | 读多写少、读操作不应被阻塞 |

### 面试题

**Q: RCU 是什么？和读写锁有什么区别？**

> RCU（Read-Copy-Update）是 Linux 内核的无锁同步机制。读者直接读取不加锁（永远不阻塞），写者复制数据修改后原子替换指针，旧版本等待宽限期结束后释放。
>
> **与读写锁区别**：
> | | 读写锁 | RCU |
> |------|--------|-----|
> | **读者开销** | 有（获取/释放锁） | **零**（仅禁止抢占） |
> | **写者开销** | 高（等待读者） | 中（复制 + 等待宽限期） |
> | **读者阻塞写者** | 是 | **否** |
> | **适用场景** | 临界区较长 | **读极多写极少** |
>
> Linux 内核大量使用 RCU（路由缓存、dentry cache、模块管理），是支撑内核高性能和可扩展性的关键技术。

## 5.14 内存屏障与 Go 内存模型 ★★★★

### CPU 指令重排

CPU 和编译器为了性能会**重排指令顺序**，但重排对单线程透明，对多线程可能是灾难：

```go
// 线程1:
a = 1        // ① 写 a
flag = 1     // ② 写 flag（可能被 CPU 重排到 ① 前面！）

// 线程2:
if flag == 1 {
    print(a) // 本应打印 1，但可能打印 0！
             // 因为 a=1 还没有被线程2看到
}
```

### 内存屏障（Memory Barrier/Fence）

内存屏障是一条指令，强制 CPU 在屏障处**完成所有之前的访存操作**，屏障前后的指令不能跨越屏障重排。

| 类型 | 作用 |
|------|------|
| **StoreStore 屏障** | 前面的写必须在后面的写之前完成 |
| **LoadLoad 屏障** | 前面的读必须在后面的读之前完成 |
| **StoreLoad 屏障** | 前面的写必须在后面的读之前完成（最重，x86 不天然支持） |
| **LoadStore 屏障** | 前面的读必须在后面的写之前完成 |

### Go happens-before 规则

Go 内存模型定义了 happens-before 关系，保证多 goroutine 之间的可见性：

```
1. Channel：写操作 happens-before 对应的读操作
   ch <- data
   <- ch      // 一定能看到 data 的所有写入

2. sync.Mutex：Unlock happens-before 下一次 Lock
   mu.Unlock()
   mu.Lock()  // 一定能看到 Unlock 之前的所有操作

3. sync.RWMutex：RLock 能看到上一次 RUnlock 后的所有写

4. Once：once.Do(f) 中的 f 的完成 happens-before 任何 once.Do 的返回

5. 原子操作：atomic.Store / atomic.Load 提供顺序一致性

6. goroutine 创建：go func() 中能看到 go 语句之前的所有操作
7. goroutine 结束：goroutine 的退出不保证对其他 goroutine 可见！
   → 必须用 channel/sync.WaitGroup 等显式同步
```

```go
// 典型错误：以为 goroutine 结束就可见
var data string
go func() {
    data = "hello"  // 在 goroutine 内修改
}()
// ❌ 这里不一定能看到 data = "hello"！
// goroutine 退出没有 happens-before 关系
// → 必须用 sync.WaitGroup 或 channel 同步
```

### 面试题

**Q: Go 中如何保证多 goroutine 之间的数据可见？**

> 必须通过**同步原语**建立 happens-before 关系：channel 通信、sync.Mutex 的 Lock/Unlock、sync.WaitGroup、sync/atomic、sync.Once。直接共享读写没有 happens-before 保证，`go test -race` 可检测数据竞争。
>
> **核心原则**："不要通过共享内存通信，而要通过通信共享内存"——channel 是 Go 推荐的同步方式。

## 5.15 经典同步问题 ★★★★

### 哲学家进餐问题

5 个哲学家围坐，5 根筷子，每人需要**左右两根**筷子才能吃饭。如果所有人同时拿左边筷子 → 死锁。

**解决方案**：

**方案一：最多 4 人同时拿筷子**

```go
sem := make(chan struct{}, 4) // 信号量限制并发数
for i := 0; i < 5; i++ {
    go func(id int) {
        for {
            sem <- struct{}{}        // 最多 4 人同时尝试拿筷子
            left.Lock()              // 拿左边
            right.Lock()             // 拿右边
            // 吃饭...
            right.Unlock()
            left.Unlock()
            <-sem
        }
    }(i)
}
```

**方案二：奇偶编号取不同顺序**（破坏循环等待）

```go
for i := 0; i < 5; i++ {
    go func(id int) {
        for {
            if id%2 == 0 {
                left.Lock()   // 偶数先左后右
                right.Lock()
            } else {
                right.Lock()  // 奇数先右后左
                left.Lock()
            }
            // 吃饭...
            right.Unlock()
            left.Unlock()
        }
    }(i)
}
```

**方案三：同时拿两根**（用额外的互斥锁保护拿筷子的动作，使其原子化）

### 生产者-消费者问题

```
有限缓冲区，生产者放数据，消费者取数据：
  - 缓冲区满 → 生产者等待
  - 缓冲区空 → 消费者等待
```

**信号量解法**：

```
empty = N（空槽位数）
full = 0（满槽位数）
mutex = 1（互斥访问缓冲区）

生产者:                    消费者:
  P(empty)    等空位         P(full)     等数据
  P(mutex)    进临界区       P(mutex)    进临界区
  放入数据                   取出数据
  V(mutex)    出临界区       V(mutex)    出临界区
  V(full)     通知有数据     V(empty)    通知有空位
```

**Go 中直接使用 channel**：

```go
ch := make(chan int, 10) // 缓冲区大小 = 10
// 生产者
go func() {
    for i := 0; ; i++ {
        ch <- i  // 满时自动阻塞
    }
}()
// 消费者
for v := range ch {
    fmt.Println(v) // 空时自动阻塞
}
```

### 面试题

**Q: 哲学家进餐问题如何解决死锁？**

> 三个方案：① 限制最多 N-1 人同时拿筷子（信号量限制并发）；② 奇偶编号拿不同方向（破坏循环等待）；③ 把"拿两根筷子"做成原子操作。核心都是**破坏死锁四条件之一**。

# 第六章 死锁 ★★★★★

## 6.1 什么是死锁

死锁（Deadlock）是指两个或多个进程/线程**互相等待对方持有的资源**，导致所有相关方都无法继续执行的状态。

```
线程A 持有 锁1，等待 锁2
线程B 持有 锁2，等待 锁1
→ 死锁！两者永远等待
```

---

## 6.2 死锁产生条件 ★★★★★

四个必要条件，**缺一不可**：

| 条件 | 说明 |
|------|------|
| **1. 互斥** | 资源一次只能被一个进程使用 |
| **2. 请求保持** | 进程已持有资源，又请求新资源 |
| **3. 不可剥夺** | 进程持有的资源不能被强制夺走 |
| **4. 循环等待** | 存在进程-资源的循环等待链 P1→P2→...→Pn→P1 |

### 面试题

**Q: 死锁的必要条件是什么？**

> 四个条件必须**同时满足**才会死锁：
> 1. **互斥访问**：资源不能共享
> 2. **持有并等待**：占着一些资源，同时等待另一些资源
> 3. **不可剥夺**：已获得的资源不能被强行抢走
> 4. **循环等待**：形成循环等待链
>
> 破坏**任意一个**条件就可以预防死锁。

---

## 6.3 死锁预防（破坏四个条件之一）

| 破坏条件 | 方法 | 缺点 |
|---------|------|------|
| **破坏互斥** | 资源共享化（如无锁数据结构） | 不是所有资源都能共享 |
| **破坏请求保持** | 一次性申请所有资源 | 资源利用率低，可能饥饿 |
| **破坏不可剥夺** | 申请失败则释放已持有的资源 | 实现复杂，可能活锁 |
| **破坏循环等待** | 所有资源编号，按顺序申请 | 需要提前知道所有资源 |

---

## 6.4 死锁避免

### 银行家算法

操作系统**动态检查**每次资源分配是否会导致死锁：

```
当进程申请资源时，系统判断：
如果分配后，系统仍处于"安全状态" → 分配
否则 → 拒绝分配，进程等待
```

**安全状态**：存在一种进程执行序列，使得每个进程都能获得所需资源并顺利执行完毕。

**缺点**：
- 需要预知每个进程的最大资源需求
- 算法复杂度高 O(n² × m)，不常用于实际系统

---

## 6.5 死锁检测

系统允许死锁发生，但通过**周期性检测**资源分配图是否有环来判断死锁。

- **检测周期**：取决于死锁发生频率和受影响进程数
- 可以用 **wait-for 图**（资源分配图的简化版）来检测

---

## 6.6 死锁解除

检测到死锁后的处理方法：

1. **剥夺资源**：从一些进程中抢占资源，分配给其他进程
2. **终止进程**：
   - 一次性终止所有死锁进程（成本高）
   - 逐个终止，直到死锁解除（每次重新检测）
3. **回滚**：利用检查点回滚进程到死锁前的状态

### 面试题

**Q: 如何解决死锁？**

> 四个层次：
>
> 1. **预防**：破坏四个必要条件之一（如资源排序申请）
> 2. **避免**：银行家算法，分配前检查是否安全
> 3. **检测**：周期性检测资源分配图是否有环
> 4. **解除**：检测到后 → 终止进程 / 剥夺资源 / 回滚
>
> **实际开发中最常用的方法**：
> - **按顺序加锁**（破坏循环等待）
> - **设置超时**（tryLock 带超时，破坏请求保持）
> - **一次性申请所有锁**
> - **Go 中注意 channel 操作不要成环**

# 第七章 内存管理 ★★★★★

## 7.1 内存管理基础

### OS 整体内存管理机制概述

操作系统内存管理是一个**多层次**的体系，从硬件到应用层逐级抽象：

```
应用程序
    │  malloc / free (glibc ptmalloc)
    ▼
用户态内存分配器（ptmalloc / jemalloc / tcmalloc）
    │  brk / mmap 系统调用
    ▼
虚拟内存管理（内核）
    │  页表、缺页处理、页面置换
    ▼
物理内存管理（内核）
    │  伙伴系统 Buddy（分配连续物理页框）
    │  Slab 分配器（管理内核小对象）
    ▼
硬件（MMU + TLB + 物理内存 + 磁盘 Swap）
```

**OS 内存管理的核心职责**：

| 职责 | 说明 | 实现机制 |
|------|------|---------|
| **地址转换** | 虚拟地址 → 物理地址 | MMU + 多级页表 + TLB |
| **内存分配** | 为进程/内核分配物理内存 | 伙伴系统 + Slab |
| **内存保护** | 进程间隔离，防止越界访问 | 页表权限位（R/W/U），段界限 |
| **内存扩展** | 让进程使用超过物理内存的地址空间 | 虚拟内存 + Swap 换入换出 |
| **内存共享** | 多进程共享同一物理内存 | 页表映射同一物理页框（共享库、mmap） |
| **内存回收** | 物理内存不足时回收页面 | kswapd 回收 + OOM Killer 兜底 |

### 为什么需要内存管理

1. **多道程序**：多个进程同时驻留内存，需要分配不同的区域
2. **内存有限**：物理内存远小于所有进程所需的总内存 → 虚拟内存
3. **隔离保护**：进程 A 不能访问进程 B 的内存
4. **共享**：有些内存需要进程间共享（如共享库 .so/.dll）
5. **碎片管理**：分配/释放过程中产生碎片，需要高效管理（伙伴系统处理外部碎片、Slab 处理内部碎片）

---

## 7.2 虚拟内存 ★★★★★

### 什么是虚拟内存

虚拟内存（Virtual Memory）是一种内存管理技术，它让每个进程以为**自己独占全部内存空间**（如 32 位 4GB），而实际上物理内存远小于此，不足的部分用磁盘来"冒充"。

```
虚拟地址 ──[MMU/页表]──→ 物理地址
   ↓                        ↓
 进程看到的         真实的物理内存
 (连续、完整)        (可能分散、部分在磁盘)
```

### 为什么需要虚拟内存

1. **让进程以为有"大内存"**：32 位进程有 4GB 地址空间，即使物理内存只有 512MB
2. **隔离性**：每个进程独立的地址空间，互不影响
3. **简化编程**：程序员只需关心连续的虚拟地址，不需要管物理内存碎片
4. **共享内存**：不同进程的虚拟页可以映射到同一物理页（如共享库）
5. **内存保护**：通过页表项的保护位（读/写/执行权限）

### 面试题

**Q: 什么是虚拟内存？**

> 虚拟内存是操作系统提供的内存抽象。每个进程看到的是**连续的、完整的虚拟地址空间**，而实际物理内存是**离散的、分页的、可能不足的**。CPU 通过 **MMU（内存管理单元）+ 页表** 将虚拟地址翻译为物理地址。不在物理内存中的页面存储在磁盘的**交换区（swap）**，访问时触发缺页异常调入内存。
>
> **核心好处**：隔离、扩展（逻辑 > 物理）、简化编程、内存保护。

## 7.3 分页机制 ★★★★★

### 页、页表、地址转换

**概念：**
- **页（Page）**：虚拟地址空间的固定大小块（通常 4KB）
- **页框（Page Frame）**：物理内存中同样大小的块
- **页表（Page Table）**：数组，索引 = 虚拟页号，值 = 物理页框号 + 标志位

**地址转换过程（以 32 位系统为例）：**

```
虚拟地址: | 页号(20位) | 页内偏移(12位) |
                     │
                     ▼
             ┌──────────────┐
             │   页表        │
             │ 页号 → 页框号 │
             └──────────────┘
                     │
                     ▼
物理地址: | 页框号(20位) | 页内偏移(12位) |
```

**页表项 Flags：**

| 标志位 | 含义 |
|--------|------|
| Present (P) | 页面是否在物理内存中 |
| Read/Write (R/W) | 是否可写 |
| User/Supervisor (U/S) | 用户态是否可访问 |
| Accessed (A) | 页面是否被访问过（用于 LRU） |
| Dirty (D) | 页面是否被修改过（换出时需写磁盘） |

### 面试题

**Q: 虚拟地址如何转换为物理地址？**

> 1. 虚拟地址 = **虚拟页号 VPN** + **页内偏移**
> 2. CPU 先查 **TLB**（快表），命中直接得到物理页框号
> 3. TLB 未命中 → 查**页表**（多级页表逐级查）
> 4. 页表基址在 CR3 寄存器中
> 5. 从页表中获取**物理页框号 PFN**
> 6. 物理地址 = **PFN + 页内偏移**（页内偏移不变）
> 7. 如果页表项 Present=0 → **缺页异常** → 从磁盘调入

## 7.4 多级页表 ★★★★★

### 二级页表

**问题**：单级页表太大了！32 位系统，4KB 页面 → 每个进程需要 2^20 = 1,048,576 个页表项，每个 4 字节 → **4MB 页表**。每个进程一份 → 内存爆炸。

**解决：二级页表**

```
虚拟地址: | 页目录索引(10位) | 页表索引(10位) | 页内偏移(12位) |
```

- **页目录**：指向二级页表，1024 个条目
- **二级页表**：指向实际物理页面
- **只分配实际使用的页表项** → 绝大多数不用的地址不需要页表空间

64 位系统一般使用**四级页表**（PGD → PUD → PMD → PTE）。

### 面试题

**Q: 为什么需要多级页表？**

> 1. **节省内存**：单级页表每个进程需要 4MB（32位），多级只用分配实际使用的部分。未使用的虚拟地址区域不需要对应的页表内存
> 2. **稀疏地址空间友好**：进程地址空间大部分是空洞（如堆和栈之间巨大空白），多级页表不需要为这些空洞分配页表
> 3. **按需分配**：页表本身也可以被换出到磁盘
>
> **代价**：地址转换需要多次访存（二级需要 2 次，四级需要 4 次）→ 所以需要 TLB 加速。
>
> **注意**：32 位 Linux 使用二级页表（PGD + PTE），64 位 Linux 使用四级页表（PGD→PUD→PMD→PTE）。

---

## 7.5 TLB ★★★★★

### 快表

TLB（Translation Lookaside Buffer，页表缓存/快表）是 **MMU 内部的高速缓存**，用于缓存最近使用的**虚拟页号 → 物理页框号**的映射关系。

```
CPU 发出虚拟地址
    │
    ▼
┌──────┐
│ TLB  │ ← 命中：几个 CPU 时钟周期 → 直接得到物理地址
└──────┘
    │ 未命中（TLB Miss）
    ▼
┌──────┐
│ 页表  │ ← 多次访存（4级页表需4次）
└──────┘
    │
    ▼
将结果加载到 TLB
```

**为什么 TLB 有效？** 程序的**时间局部性**和**空间局部性**。

### 面试题

**Q: TLB 是什么？**

> TLB（Translation Lookaside Buffer，快表/页表缓存）是 CPU 内部 MMU 中的硬件缓存，用来缓存**虚拟地址到物理地址的映射**。TLB 命中时地址转换只需几个 CPU 时钟周期；TLB 未命中（Miss）则需要多次访存查多级页表。
>
> **进程切换时需要刷新 TLB**（不同进程有不同的地址空间），这是进程切换的隐性开销之一（现代 CPU 通过 PCID/ASID 部分缓解）。
>
> **TLB 与 CPU Cache 的关系**：
> - 虚拟地址 → TLB 转物理地址 → 再去 L1/L2/L3 Cache 找数据
> - 有些 CPU 使用 VIPT（虚拟索引物理标记）Cache，可以同时查询 TLB 和 Cache

## 7.6 页面置换算法 ★★★★★

当物理内存满时，需要**换出**（swap out）一些页面到磁盘，腾出空间给新页面。选择哪一个页面换出，就是页面置换算法要解决的问题。

### FIFO（First In First Out）

- 选择**最早进入内存**的页面淘汰
- 简单，但可能淘汰频繁使用的页面（Belady 异常：有时增加内存反而缺页更多）

### LRU（Least Recently Used，最近最少使用）

- 选择**最长时间没被访问**的页面淘汰
- 基于局部性原理，效果好，但纯硬件实现复杂
- **实现方式**：链表（每次访问移到头部，换出尾部）→ O(1) 但需硬件支持
- **近似 LRU**：CLOCK 算法

### LFU（Least Frequently Used，最不经常使用）

- 选择**使用次数最少**的页面淘汰
- 缺点：新页面使用次数少容易被误淘汰

### CLOCK（时钟算法，近似 LRU）

Linux 实际采用的**改进版 CLOCK 算法**：

```
维护一个环形链表 + 访问位（Access Bit）
指针循环扫描：
  - 访问位 = 1 → 清 0，跳过
  - 访问位 = 0 → 淘汰该页
```

这是 LRU 的**高效近似**，不需要维护完整的访问顺序。

### 面试题

**Q: LRU 原理是什么？**

> LRU（Least Recently Used）选择**最久未使用的页面**淘汰。基于局部性原理：最近使用过的页面很可能再次使用，最久没使用的被换出的代价最小。
>
> **实现**：
> - **理想版**：双向链表 + HashMap，访问移到头部，换出尾部 → O(1)，内存开销大
> - **实际版**：CLOCK 算法（环形链表 + 访问位），近似 LRU，硬件支持好
>
> **Linux 实现**：改进版 CLOCK（双指针，区分是否被修改过），优先换出"未访问 + 未修改"的页面。

## 7.7 缺页异常 ★★★★★

### Page Fault 的三种类型

**缺页异常（Page Fault）** 不只是"页面不在内存中"这么简单，实际上有三种类型：

| 类型 | 场景 | 处理 | 性能影响 |
|------|------|------|---------|
| **Minor Page Fault（次缺页）** | 页已在物理内存中，但页表项未建立映射（如 COW 后的写保护页、mmap 已加载但未关联） | 只需在页表中建立映射，**不读磁盘** | 极快（~几 μs） |
| **Major Page Fault（主缺页）** | 页面不在物理内存中，需要从磁盘读取 | 分配物理页框 + 磁盘 I/O | 慢（~几 ms，受磁盘影响） |
| **Invalid Page Fault（无效缺页）** | 访问非法地址（如空指针、野指针） | 内核发送 **SIGSEGV** 信号终止进程 | N/A |

> **性能排查关键指标**：`vmstat` 的 `majflt` 列就是 Major Page Fault 计数。大量 Major Fault 说明物理内存不足或频繁 mmap I/O。`perf stat -e page-faults` 可查看进程缺页总次数。

### Major Page Fault 处理流程

当 CPU 访问的虚拟地址对应的页表项 **Present = 0**，且页面确实不在物理内存中时：

```
1. CPU 发现 Present=0 → 触发缺页异常（Page Fault）
2. 硬件保存现场 → 陷入内核
3. 内核检查该地址是否合法（是否在进程的 VMA 范围内）
4. 非法 → 发送 SIGSEGV 信号（Segmentation Fault）
5. 合法 → 缺页处理：
   a. 找空闲物理页框（没有则执行页面置换）
   b. 从磁盘读入数据：
      - 代码/文件映射页 → 从文件读取
      - 匿名页 → 从 swap 分区读入
      - 未初始化页 → 分配零页（Zero Page）
   c. 更新页表项（填入物理页框号，设置 Present=1）
6. 重新执行触发异常的指令
```

### 面试题

**Q: 缺页异常发生了什么？有哪几种类型？**

> 缺页异常有三种：
> 1. **Minor Page Fault（次缺页）**：页已在内存但映射未建立（COW 写保护、mmap 延迟映射），只需建立页表映射，不读磁盘，极快
> 2. **Major Page Fault（主缺页）**：页不在内存中，需从磁盘读入（swap/文件 I/O），慢（ms 级）
> 3. **Invalid Page Fault**：非法地址 → SIGSEGV
>
> **处理流程**：
> 1. 访问虚拟地址 → 页表项 Present=0 → CPU 触发 Page Fault
> 2. 陷入内核 → 内核检查地址合法性
> 3. 找空闲物理页框（可能触发页面置换）
> 4. 从磁盘读入页面数据（swap/文件映射）
> 5. 更新页表，设置 Present=1
> 6. 返回用户态，重新执行被中断的指令
> 7. **对进程透明**：除了变慢，进程不知道发生了缺页
>
> **面试要点**：应用启动慢、频繁 GC、大内存扫描 → 关注 Major Page Fault。`vmstat` 的 `majflt` 列是关键排查指标。

## 7.8 Swap 交换机制 ★★★★

### 什么是 Swap

Swap（交换）是当物理内存不足时，操作系统将**暂时不用的内存页**换出到磁盘（swap 分区或 swap 文件），腾出物理内存给更需要的进程使用。当被换出的页再次被访问时，触发 Major Page Fault 将其换回。

```
物理内存（RAM）                   磁盘（Swap）
┌──────────────┐    换出(Swap Out)   ┌──────────────┐
│  活跃页面     │ ─────────────────→  │ 不活跃页面    │
│              │ ←─────────────────  │              │
│  新分配页面   │    换入(Swap In)    │              │
└──────────────┘                    └──────────────┘
```

### swappiness 参数

```bash
cat /proc/sys/vm/swappiness   # 默认 60（范围 0~100）
```

| 值 | 行为 |
|------|------|
| **0** | 尽量不用 swap，只在 OOM 前才换出（实际 3.5+ 内核会特殊情况使用） |
| **60**（默认） | 平衡使用，内存压力适中时开始换出 |
| **100** | 激进换出，尽早使用 swap |

**数据库最佳实践**：MySQL、Redis、Elasticsearch 通常建议设置 `vm.swappiness=1`（或较低值），避免关键热数据被换出导致延迟抖动。

### Swap 分区 vs Swap 文件

| | Swap 分区 | Swap 文件 |
|------|-----------|----------|
| **性能** | 更快（跳过文件系统层） | 略慢（经过文件系统） |
| **灵活性** | 创建后难以调整大小 | 可以随时调整大小 |
| **适用场景** | 传统部署 | 云环境、容器（灵活扩缩） |
| **休眠支持** | 支持 | 不一定支持 |

### 换出策略（选择哪些页换出）

Linux 使用**双指针改进版 CLOCK 算法**，优先换出：

1. **未访问 + 未修改**（最佳候选，无需写磁盘）
2. **未访问 + 已修改**（需要先写回磁盘/swap）
3. **已访问 + 未修改**
4. **已访问 + 已修改**（最差候选）

交换范围：
- **匿名页**（堆、栈、mmap 匿名映射）：换出到 swap 分区
- **文件映射页**（代码、mmap 文件映射）：直接丢弃（需要时从文件重新读取）

### 面试题

**Q: Swap 是什么？swappiness 如何调整？**

> Swap 是内存不足时将不活跃页面换出到磁盘的机制。`swappiness`（0-100）控制内核使用 swap 的激进程度，60 为默认。数据库（MySQL/Redis）建议设为 1，避免热数据被换出导致延迟抖动。
>
> **排查**：`vmstat 1` 看 si/so 列（swap in/out），持续 > 0 说明物理内存不足。大量 Major Page Fault + si/so 持续增长 → 急需扩内存。

## 7.9 内存分段（Segmentation）

### 段表原理

按**逻辑单元**划分内存（代码段、数据段、栈段等），每个段有各自的基址和长度。早期 Intel 8086 使用段式管理（CS:IP 寻址）。

**逻辑地址 → 物理地址转换**：

```
逻辑地址:  [段选择子] | [段内偏移]
               │
               ▼
         查段表（GDT/LDT）
               │
               ▼
         段描述符中的 段基址 + 段内偏移 = 物理地址
```

**段表项（段描述符）包含**：

| 字段 | 作用 |
|------|------|
| **段基址** | 段在物理内存中的起始地址 |
| **段界限** | 段的最大偏移（大小限制） |
| **访问权限** | 读/写/执行 |
| **特权级 DPL** | Ring 0（内核）~ Ring 3（用户） |
| **存在位 P** | 段是否在内存中 |

**访问越界检测**：偏移 > 段界限 → CPU 触发**段错误（Segmentation Fault）**。

**现代做法**：纯分段已淘汰，现在是**段页式**——外层按段划分逻辑区域（代码、数据、栈各有保护），内层每段内按分页管理。Linux 主要依赖分页，分段仅做权限控制。

### 面试题

**Q: 段表原理是什么？段式地址如何转换？**

> 段式管理中，逻辑地址 = **段号 + 段内偏移**。CPU 通过段号查段表，获取段基址，物理地址 = 段基址 + 偏移。段表项还包含段界限和访问权限，偏移超出界限则触发段错误（Segmentation Fault）。现代系统使用段页式（外层分段做权限隔离，内层分页做物理管理），纯分段已被淘汰。

目前的系统基本使用**段页式**：外层按段划分（给不同区域设置保护），内层段内分页管理。

---

## 7.10 分页与分段区别 ★★★★

| 维度 | 分页 | 分段 |
|------|------|------|
| **划分依据** | 固定大小（物理） | 逻辑单元（代码/数据/栈） |
| **大小** | 固定（4KB/2MB/1GB） | 可变（由程序决定） |
| **程序员感知** | 透明 | 可见（段） |
| **目的** | 虚拟内存管理 | 逻辑组织 + 保护 |
| **碎片** | 内部碎片（最后一页用不完） | 外部碎片（难找连续的段） |
| **地址** | 一维（虚拟页号） | 二维（段号 + 段内偏移） |

# 第八章 Linux 内存管理 ★★★★★

## 8.1 Linux 内存布局

进程虚拟地址空间（以 32 位为例，3G/1G 划分）：

```
高地址
┌────────────┐ 0xC0000000
│   内核空间   │ (1GB，所有进程共享)
├────────────┤ 0xBF??????
│    栈      │ ↓ 向下增长（局部变量）
│    ...     │
│    ...     │
│    堆      │ ↑ 向上增长（malloc）
├────────────┤ (program break)
│   BSS 段   │ 未初始化全局变量（.bss）
├────────────┤
│   数据段    │ 已初始化全局变量（.data）
├────────────┤
│   代码段    │ 程序的机器指令（.text）
├────────────┤
│   保留区    │ 防止空指针解引用
└────────────┘ 0x00000000
低地址
```

**64 位**：用户空间 128TB（0x0000... ~ 0x0000 7FFF FFFF FFFF），内核空间 128TB。

### 面试题

**Q: 堆和栈的区别？**

> | 维度 | 栈（Stack） | 堆（Heap） |
> |------|------------|-----------|
> | **分配方式** | 编译器自动分配释放 | 程序员手动 malloc/free（Go 中 GC 管理） |
> | **大小** | 有限（~8MB，ulimit -s 查看） | 受虚拟地址空间限制（更大） |
> | **增长方向** | 从高地址向低地址 | 从低地址向高地址 |
> | **分配速度** | 极快（移动 SP 指针） | 较慢（需查找空闲块） |
> | **碎片** | 无碎片 | 有碎片（需内存管理算法） |
> | **生命周期** | 函数返回即释放 | 手动释放或 GC |
> | **存储内容** | 局部变量、返回地址、函数参数 | 动态分配的对象 |
> | **线程安全** | 每线程独立栈（天然安全） | 多线程共享堆（需同步） |

### Go 逃逸分析（Escape Analysis）

Go 编译器在编译时自动判断变量应该分配在**栈**还是**堆**，这就是**逃逸分析**：

```go
// 不逃逸 → 分配在栈上（高效）
func sum(a, b int) int {
    result := a + b   // result 不会在函数外被引用 → 栈分配
    return result
}

// 逃逸 → 分配在堆上
func newUser(name string) *User {
    u := User{Name: name}  // u 返回指针，被外部引用 → 逃逸到堆
    return &u
}
```

**逃逸的常见原因**：
1. **返回局部变量的指针**（如上例）
2. **interface{} 装箱**：具体类型赋值给 interface{} 通常导致逃逸
3. **闭包捕获**：闭包引用外部变量，该变量可能逃逸
4. **变量过大**：超过栈能容纳的大小
5. **编译器无法确定大小**：如 append 不确定的切片

```bash
# 查看逃逸分析结果
go build -gcflags="-m"    # -m=1 基础逃逸信息
go build -gcflags="-m -m" # -m=2 详细逃逸信息

# 输出示例：
# ./main.go:10:2: moved to heap: u   → u 逃逸到堆
# ./main.go:15:3: result does not escape → result 在栈上
```

**性能影响**：
- **栈变量**：函数返回时自动回收，零 GC 开销
- **堆变量**：需要 GC 扫描和回收，有一定的停顿和 CPU 开销

**面试题**：

**Q: 什么是逃逸分析？什么情况会导致逃逸？**

> Go 编译器自动判断变量分配位置。函数返回后仍被引用的变量**逃逸**到堆（需 GC 回收），无法逃逸的分配在栈（函数返回即释放）。
>
> 逃逸场景：返回局部指针、interface{} 装箱、闭包捕获、变量过大。可通过 `go build -gcflags="-m"` 查看逃逸情况。
>
> **优化建议**：减少不必要的指针返回、避免不必要的 interface{} 装箱以降低 GC 压力。

## 8.2 malloc 原理 ★★★★★

### brk 与 mmap

malloc 并不是每次调用都去向内核申请内存，而是通过两种系统调用获取内存：

**1. brk/sbrk（小内存，≤128KB）**

```c
int brk(void *addr);  // 直接设置 program break（堆顶）
void *sbrk(intptr_t increment); // 移动堆顶指针
```

- 通过移动堆顶指针（program break）来扩展堆空间
- 分配的内存在**堆区**
- 释放时不会立即归还给内核（形成内存碎片），而是被 malloc 的内存池管理
- 可能导致大量已释放但未归还的内存

**2. mmap（大内存，>128KB）**

```c
void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);
```

- 在进程地址空间的**文件映射区**（堆和栈之间）分配独立的内存区域
- 分配大块内存时使用
- 释放（munmap）后**立即归还**给操作系统

**malloc 内部实现（以 ptmalloc/glibc 为例）：**

```
malloc(size):
  if size < MMAP_THRESHOLD（默认 128KB，动态调整）: → brk/sbrk 在堆区分配
  if size >= MMAP_THRESHOLD:                        → mmap 分配匿名内存

free(ptr):
  检查 chunk header，归还到 bins/fastbins
  对于 mmap 区域 → munmap 归还给 OS
```

> **注意**：`M_MMAP_THRESHOLD` 默认 128KB，但 glibc 会**动态调整**（当 mmap 区域的 free 总大小超过阈值后自动增大），可通过 `mallopt(M_MMAP_THRESHOLD, ...)` 手动设置。

### 面试题

**Q: malloc 底层如何实现？**

> malloc 是 glibc 提供的内存分配器，有两种底层来源：
>
> 1. **brk/sbrk**：移动堆顶指针分配小内存（<128KB）。释放后不立即归还 OS 而是缓存起来，下次分配复用。内存碎片可能比较严重，但分配快。
>
> 2. **mmap**：在文件映射区分配独立内存区域，用于大内存（≥128KB）。释放（munmap）后立即归还 OS。
>
> **内部管理**：使用 bins/fastbins 管理不同大小的空闲 chunk，采用类似 SLAB 的缓存机制。实际上 Go 的 TCMalloc 思想、jemalloc 等替代方案在减少碎片和并发性能上更优。

### glibc ptmalloc 内部结构详解

ptmalloc 使用多种数据结构管理空闲内存 chunk：

| 结构 | 说明 | 特点 |
|------|------|------|
| **fastbins** | 缓存小 chunk（16~80 字节，可配） | LIFO 单链表，不合并，极快分配/释放 |
| **small bins** | 管理 512 字节以下的 chunk | 64 个双向链表，按固定大小分档 |
| **large bins** | 管理 512 字节以上的 chunk | 63 个双向链表，按范围分档 |
| **unsorted bin** | 新释放的 chunk 暂存区 | 类似缓冲区，后续分配时分类到 small/large bins |
| **tcache**（glibc 2.26+） | **线程本地缓存** | 每个线程独立的缓存，减少锁竞争，类似 Go mcache 的思想 |

```
malloc 分配路径：
  1. 查 tcache → 命中直接返回（无锁，最快）
  2. 查 fastbins / small bins → 命中返回
  3. 查 unsorted bin → 命中后分类返回
  4. 查 large bins → 找到最合适的
  5. 扩展堆顶（brk）或 mmap 新内存
```

> **Go 分配器对比**：Go 基于 TCMalloc 思想，使用 **mcache（P级，无锁）→ mcentral → mheap（全局，加锁）** 三级缓存，为每个 P 分配独立的 mcache，避免锁竞争。这与 glibc 的 tcache 设计思路一致。

## 8.3 mmap 原理 ★★★★★

### 文件映射

mmap 可以将文件内容直接映射到进程的虚拟地址空间：

```c
// 文件映射：读写文件就像读写内存
char *p = mmap(NULL, size, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
p[0] = 'X';  // 直接修改，脏页由内核写回文件
```

**原理**：
- 建立虚拟地址 → 文件的页表映射（页表项指向文件的磁盘位置）
- 第一次访问 → 缺页异常 → 从文件读入 Page Cache → 建立物理页映射
- 修改数据 → 标记为脏页 → 内核异步写回文件

### 零拷贝

mmap + write 实现零拷贝（减少一次内核拷贝）。详见 8.4。

### 面试题

**Q: mmap 为什么快？**

> 1. **省略一次拷贝**：传统 read+write 需要 4 次拷贝（磁盘→内核→用户→内核→网卡），mmap+write 只需要 3 次（磁盘→内核→内核→网卡），省去了内核→用户的拷贝
> 2. **直接访问 Page Cache**：mmap 映射文件后，读写直接操作 Page Cache 中的物理内存页，不需要系统调用
> 3. **减少系统调用开销**：对文件的大量随机访问只需一次 mmap 调用，后续都是内存操作
> 4. **操作系统负责脏页写回**：修改自动被内核写回文件，不用手动 write
>
> **注意**：mmap 也有开销——缺页异常和 TLB miss 增加，对小文件、顺序读写场景不一定比 read/write 快。

## 8.4 零拷贝 ★★★★★

### 传统数据传输的问题

传统 `read() + write()` 将文件发送到网络：

```
磁盘 → 内核缓冲区(read buffer) → 用户缓冲区 → 内核缓冲区(socket buffer) → 网卡
         (1) DMA拷贝              (2) CPU拷贝       (3) CPU拷贝              (4) DMA拷贝
```

**4 次拷贝，2 次 CPU 拷贝，4 次上下文切换（read+write 各 2 次）。**

### mmap + write

```
磁盘 → 内核缓冲区(Page Cache) → 内核缓冲区(socket buffer) → 网卡
         (1) DMA                   (2) CPU拷贝                (3) DMA
```

**3 次拷贝，1 次 CPU 拷贝**（省去内核→用户的拷贝）。

### sendfile（Linux 2.1）

```c
ssize_t sendfile(int out_fd, int in_fd, off_t *offset, size_t count);
```

直接在内核空间传输数据：

```
磁盘 → 内核缓冲区(Page Cache) → 内核缓冲区(socket buffer) → 网卡
         (1) DMA                  (2) CPU拷贝               (3) DMA
```

**3 次拷贝，1 次 CPU 拷贝，2 次上下文切换（一次 sendfile 调用）。**

### sendfile + DMA Gather Copy（Linux 2.4+，真正的零拷贝）

```
磁盘 → 内核缓冲区(Page Cache) ──┐
         (1) DMA                 │ (2) 描述符传递(只传地址+长度,不拷贝数据!)
                                  → 网卡（DMA Gather）
                                  (3) DMA
```

**2 次 DMA 拷贝，0 次 CPU 拷贝！只有 1 次 sendfile 上下文切换！**

这就是 **"零拷贝"的真正含义：CPU 不参与数据拷贝**。

### splice

```c
ssize_t splice(int fd_in, loff_t *off_in, int fd_out, loff_t *off_out, size_t len, unsigned int flags);
```

splice 在两个文件描述符之间**移动数据**，无需经过用户空间。类似于管道，但更通用——可以实现**任意两个 fd 之间的零拷贝**（如 socket → socket）。

### DMA（Direct Memory Access，直接内存访问）

DMA 使得外设可以直接访问系统内存，**不需要 CPU 介入**来拷贝数据。CPU 只需要告诉 DMA 控制器："从磁盘 X 读到内存地址 Y，大小 Z"，然后去做别的事，DMA 完成后通过中断通知 CPU。

```
有 DMA：  磁盘 →[DMA]→ 内存   | CPU 可以做其他事
无 DMA：  磁盘 →[CPU]→ 内存   | CPU 全程参与搬运数据（PIO模式）
```

### 面试题

**Q: 什么是零拷贝？**

> 零拷贝（Zero Copy）是指数据传输过程中**避免 CPU 在内核空间和用户空间之间拷贝数据**的技术。
>
> **传统方式**（read+write）：4 次拷贝（2 次 CPU + 2 次 DMA），4 次上下文切换
>
> **零拷贝实现**：
> 1. **mmap + write**：减少 1 次 CPU 拷贝
> 2. **sendfile**：在内核完成传输，减少上下文切换
> 3. **sendfile + DMA Gather Copy**：真正的零拷贝（0 次 CPU 拷贝，只有 2 次 DMA）
> 4. **splice**：在任意两个 fd 之间零拷贝传输数据
>
> **关键**：零拷贝不是"不拷贝"，而是 **CPU 不参与数据拷贝**——所有拷贝都由 DMA 完成。

**Q: Kafka 为什么快？**

> Kafka 大量使用零拷贝技术：
> 1. **sendfile**：生产者写入的数据存储在 Page Cache 中，消费者读取时通过 sendfile 直接从 Page Cache 传输到网卡，零 CPU 拷贝
> 2. **顺序写磁盘**：Kafka 以追加方式顺序写入，利用磁盘的顺序写入性能（比随机写快 100+ 倍）
> 3. **Page Cache 利用**：数据先被 Page Cache 吸收，读操作直接命中缓存，避免物理磁盘 I/O
> 4. **mmap**：索引文件（index）通过 mmap 映射，提高随机读性能

## 8.5 伙伴系统 Buddy System ★★★

### 什么是伙伴系统

伙伴系统（Buddy System）是 Linux 内核管理**物理页框**的分配器，负责处理 `alloc_page()` / `__get_free_pages()` 这类内核内存分配请求。

**核心思想**：将物理内存划分为 2^N 个连续页框的块。

```
内存块大小：4KB(2^0), 8KB(2^1), 16KB(2^2), ..., 4MB(2^10)

每个大小级别维护一个空闲链表（free_area[]）
```

**分配流程**：
```
申请 8 个连续页(32KB = 2^3 × 4KB)：
  1. 查 2^3 的空闲链表 → 有则分配
  2. 没有 → 向上一级(2^4)请求 → 把 2^4 的块一分为二
  3. 一半分配，另一半加入 2^3 的空闲链表
  4. 如果 2^4 也没有 → 继续向上分裂
```

**释放流程**：
```
释放一个 2^3 的块：
  1. 检查其"伙伴"（相邻的同大小块）是否也空闲
  2. 是 → 两个 2^3 合并为 2^4 → 递归向上合并
  3. 否 → 加入 2^3 的空闲链表
```

**伙伴的定义**：两个块地址相邻 + 大小相同 + 起始地址对齐。

**优点**：有效解决**外部碎片**问题（通过合并）。
**缺点**：有**内部碎片**（申请 33KB 实际分配 64KB = 2^4 的块）。

### 面试题

**Q: 伙伴系统是什么？**

> 伙伴系统是 Linux 内核分配物理页框的算法。将内存分为 2^N 个页的块，分配时从合适大小的链表取，不够则向上分裂；释放时检查相邻的"伙伴"块，都空闲则合并。作用是高效管理物理内存、减少外部碎片。

## 8.6 Slab 分配器 ★★★

### 为什么需要 Slab

伙伴系统以**页**（4KB）为单位分配，对于内核频繁创建的小对象（如 task_struct、inode、dentry），每申请一个就分配一页太浪费。Slab 分配器在**伙伴系统之上**，管理内核**小对象**的分配。

```
应用程序 malloc() → glibc ptmalloc
         ↓
内核小对象      → Slab 分配器（kmalloc、kmem_cache_alloc）
         ↓
物理页框        → 伙伴系统（alloc_page）
```

### 工作原理

```
Slab 向伙伴系统申请整页（或连续多页）
  → 将页切分为多个固定大小的 object
  → 缓存这些 object，快速分配和回收
  
每个内核结构体有自己的 cache：
  - task_struct_cache
  - inode_cache
  - dentry_cache
```

**SLUB**（Linux 2.6.23+ 默认）：Slab 的简化改进版，减少元数据开销。

### 内存分配层次总结

```
用户态         malloc/free      → ptmalloc/jemalloc/tcmalloc
               new/delete       → (C++)
──────────────────────────────────────────────────
内核态小对象   kmalloc/kfree    → Slab/SLUB 分配器
内核态页级     alloc_page/free  → 伙伴系统 Buddy
──────────────────────────────────────────────────
硬件层                          → 物理内存页框
```

## 8.7 大页内存 Huge Pages ★★★

### 为什么需要大页

标准页 4KB → 1GB 内存需要 262,144 个页表条目 → TLB 只有几十到几百条目 → 大量 TLB Miss。

**大页（Huge Pages）**：使用 2MB 或 1GB 的页面代替 4KB。

```
4KB 页面：1GB → 262,144 次 TLB 查找（大量 Miss）
2MB 大页：1GB → 512 次 TLB 查找（全部命中）
1GB 巨页：1GB → 1 次 TLB 查找
```

### 透明大页 THP（Transparent Huge Pages）

Linux 内核自动管理大页，对应用透明：

```bash
# 查看 THP 状态
cat /sys/kernel/mm/transparent_hugepage/enabled
# always madvise never
```

**注意**：数据库（MySQL、Redis）通常**关闭 THP**，因为 THP 的碎片整理可能造成延迟抖动。

## 8.8 NUMA 架构 ★★★★

### 什么是 NUMA

NUMA（Non-Uniform Memory Access，非一致性内存访问）是多路服务器上的内存架构。与传统的 **UMA（Uniform Memory Access）** 不同，在 NUMA 架构下：

- 每个 CPU（或一组 CPU）有**本地内存**，访问速度快
- 访问**远端内存**（其他 CPU 的本地内存）需要经过互联总线（如 Intel QPI/UPI、AMD Infinity Fabric），延迟更高

```
┌─────────────────────┐     ┌─────────────────────┐
│   Node 0             │     │   Node 1             │
│  ┌──────┐ ┌───────┐ │     │  ┌──────┐ ┌───────┐ │
│  │CPU 0 │ │CPU 1  │ │←───→│  │CPU 2 │ │CPU 3  │ │
│  └──────┘ └───────┘ │ QPI │  └──────┘ └───────┘ │
│  ┌──────────────┐   │     │  ┌──────────────┐   │
│  │  本地内存     │   │     │  │  本地内存     │   │
│  │  (快 ~100ns) │   │     │  │  (快 ~100ns) │   │
│  └──────────────┘   │     │  └──────────────┘   │
│        ↑ 远端访问 ~150ns+ ↑                      │
└─────────────────────┘     └─────────────────────┘
```

### 关键概念

| 概念 | 说明 |
|------|------|
| **NUMA Node** | 一个 CPU + 本地内存的集合 |
| **本地内存** | 与 CPU 同在一个 Node 的内存，访问延迟低（~100ns） |
| **远端内存** | 其他 Node 的内存，访问延迟高（~150ns+，取决于距离） |
| **Node Distance** | Node 之间的距离（跳数），`numactl --hardware` 查看 |

### NUMA 带来的问题

1. **内存访问延迟不对等**：进程分配了远端内存 → 频繁访问时性能下降 30-50%
2. **内存分配策略不当**：进程在 Node0 运行但所有内存分配在 Node1 → 每个访问都是远端
3. **跨 Node 的锁竞争**：多个线程在不同 Node 访问同一 Cache Line → 跨 Node 的缓存一致性开销巨大

### Linux NUMA 内存策略

```bash
# 查看 NUMA 拓扑
numactl --hardware
lstopo                    # 图形化查看
numastat                  # 各 Node 内存使用统计

# 绑定进程到指定 Node
numactl --cpunodebind=0 --membind=0 ./myapp  # CPU + 内存都绑在 Node 0
numactl --interleave=all ./myapp              # 内存在所有 Node 间轮转分配
```

| 策略 | 含义 |
|------|------|
| **默认（localalloc）** | 在触发分配的 CPU 所在 Node 分配内存 |
| **membind** | 严格绑定到指定 Node |
| **preferred** | 优先从指定 Node 分配，不够再用其他 |
| **interleave** | 在所有 Node 间轮转分配（避免单 Node 过载） |

### Go 与 NUMA

Go runtime 对 NUMA 有部分感知，但调度器默认不严格做 NUMA 绑定：

```bash
# 查看 Go 进程的 NUMA 分布
numastat -p <pid>
```

**常见 NUMA 优化**：
- `GOMAXPROCS` 不超过单个 NUMA Node 的 CPU 数时可以绑定到单 Node
- 对大内存数据库/缓存（如 Redis）使用 `numactl --interleave=all` 避免单 Node 内存过载
- Docker/K8s 中通过 cpuset 限制容器只使用某个 NUMA Node

### 面试题

**Q: NUMA 是什么？为什么重要？**

> NUMA（Non-Uniform Memory Access）是多路服务器架构，CPU 访问本地内存快、访问远端内存慢。问题：进程分配了远端内存后性能下降 30-50%。
>
> 优化：绑定进程到单 NUMA Node（`numactl --membind`）、跨 Node 轮转分配（`interleave`）、Go 中控制 `GOMAXPROCS` 使其不超过单 Node 核心数。

## 8.9 OOM Killer ★★★★★

### 什么是 OOM Killer

当系统物理内存 + swap 都耗尽时，内核无法满足内存请求。此时 **OOM Killer（Out Of Memory Killer）** 会被触发，**选择一个进程杀死**来释放内存。

### 选择逻辑

每个进程有一个 `oom_score`（0~1000，越高越容易被杀）：

```
oom_score = (进程内存使用占比) × (一些调整因子)
最终分数 = oom_score + oom_score_adj
```

影响因素：
- 使用内存越多 → oom_score 越高
- root 进程 → 分数打折
- 刚启动的进程 → 分数打折（防止一直被杀重启）
- `oom_score_adj` 可手动调整：`echo -1000 > /proc/<pid>/oom_score_adj`（-1000 = 永不杀）

### 日志查看

```bash
dmesg | grep -i "killed process"
# Out of memory: Killed process 1234 (java)
# 或查看 /var/log/messages /var/log/syslog
```

### 预防措施

1. 预留足够物理内存 + swap
2. 关键服务调整 `oom_score_adj` 降低被杀概率
3. 设置内存限制（cgroup memory.limit）
4. 监控内存使用趋势，提前告警

### 面试题

**Q: OOM Killer 是什么？它如何选择要杀的进程？**

> OOM Killer 是 Linux 在内存耗尽时强制杀进程的机制。根据 `oom_score`（进程内存使用量 + 调整因子）选择分数最高的进程杀死，以释放内存。
>
> 生产环境预防：预留内存+swap、关键进程调 oom_score_adj、使用 cgroup 限制容器内存、监控告警。

# 第九章 文件系统 ★★★★

## 9.1 文件系统基础

### 文件
- 数据的有名集合，持久化存储在磁盘上
- 属性：文件名、大小、创建时间、权限、所有者

### 目录
- 包含多个文件的集合，形成树形结构
- 目录本身也是文件（存储的是目录项列表）

### inode

inode（Index Node，索引节点）是文件系统中存储文件**元数据**的结构。每个文件有且只有一个 inode。

**inode 包含：**

| 内容 | 说明 |
|------|------|
| 文件类型 | 普通文件、目录、符号链接、设备文件等 |
| 权限 | rwx（owner + group + other） |
| 硬链接数 | 指向该 inode 的目录项数 |
| 所有者 | UID、GID |
| 大小 | 文件字节数 |
| 时间戳 | atime（访问）、mtime（修改）、ctime（状态改变） |
| 数据块指针 | 指向实际数据块的指针 |

**inode 不包含的内容**：文件名！文件名存在目录的目录项中。

### 面试题

**Q: inode 是什么？**

> inode（索引节点）是文件系统中存储文件**元数据**的数据结构，每个文件有唯一 inode。它存储了文件的权限、大小、时间戳、数据块指针等元信息，但**不包含文件名**（文件名存在目录的目录项中）。
>
> **硬链接**：多个文件名 → 同一个 inode（同一文件多个名字），删除一个链接不影响数据。
> **软链接**：一个文件（有自己的 inode）→ 存储另一个文件的路径，类似快捷方式。

---

## 9.2 Linux 文件系统

| 文件系统 | 特点 |
|---------|------|
| **ext4** | 最广泛使用的 Linux 文件系统，稳定、可靠，支持最大 1EB，单文件 16TB |
| **XFS** | 高性能 64 位日志文件系统，擅长处理大文件和高并发，CentOS 7+ 默认 |
| **btrfs** | 现代 CoW（写时复制）文件系统，支持快照、压缩、子卷 |
| **tmpfs** | 基于内存的文件系统，重启后数据丢失（用于 /tmp, /dev/shm） |

### 文件系统磁盘布局

```
磁盘布局：
┌──────────┬────────────┬──────────────┬──────────────────────┐
│ 引导块    │  超级块     │   inode 表   │      数据块          │
│ (Boot)   │ Superblock │  Inode Table │    Data Blocks       │
└──────────┴────────────┴──────────────┴──────────────────────┘

- 引导块：操作系统启动引导代码
- 超级块：文件系统元数据（类型、总块数、空闲块数、inode总数）
- inode 表：所有 inode 的数组（每个文件一个 inode）
- 数据块：存储实际文件内容 + 目录条目（文件名 → inode 号映射）
```

### 文件读取全过程（路径解析）

```
open("/home/user/file.txt", O_RDONLY) 的完整过程：

1. 从根目录 inode（通常是 inode 2）开始
2. 读根目录的数据块 → 查找 "home" 目录项 → 得到 home 的 inode 号
3. 读 home 的 inode → 找到 home 的数据块 → 查找 "user" → 得到 user 的 inode
4. 读 user 的 inode → 找到 user 的数据块 → 查找 "file.txt" → 得到目标 inode
5. 权限检查通过 → 在进程的 fd 表中创建条目 → 返回 fd（整数）
6. 沿途的 dentry 和 inode 被缓存到 dentry cache 和 inode cache 加速下次查找
```

---

## 9.3 文件读写流程

```
open("/path/file", O_RDONLY)
    │
    ▼
1. 路径解析：逐级查找目录项（利用 dentry cache 加速）
2. 找到目标文件的 dentry → 获取 inode 号
3. 从磁盘读取 inode → 加载到内存（inode cache）
4. 创建 file 对象 → 加入进程的文件描述符表
5. 返回 fd

read(fd, buf, count)
    │
    ▼
1. fd → file 对象 → 获取当前文件偏移
2. 根据 inode 的数据块指针计算要读的磁盘块号
3. 先查 Page Cache，命中直接返回
4. 未命中 → 从磁盘读入 Page Cache
5. 将数据从 Page Cache 拷贝到用户态 buf

write(fd, buf, count)
    │
    ▼
1. 将数据从用户空间拷贝到 Page Cache（写缓存）
2. 标记该页为脏（Dirty）
3. 后台线程（pdflush/bdflush）定期将脏页刷入磁盘
4. close 时不保证已落盘 → 需要 fsync/fdatasync 强制刷盘
```

### 文件描述符与多进程共享

```
进程 A                      内核                      inode 表
fd 表:                 打开文件表:
fd 0 → ────────────→ [file0: pos=0, flags]──────→ inode(stdin)
fd 1 → ────────────→ [file1: pos=0, flags]──────→ inode(stdout)
fd 3 → ────────────→ [file2: pos=128, O_RDONLY]─→ inode(file.txt)
                                                         ↑
进程 B (fork 后的子进程，或独立进程)                      │
fd 表:                                                   │
fd 3 → ────────────→ [file3: pos=0, O_RDONLY]───────────┘
```

**关键理解**：

- **fork 后**：子进程复制父进程的文件描述符表，**fd 指向内核中同一个 file 对象** → 共享文件偏移量。父子同时写会交错
- **独立 open**：不同进程各自 `open` 同一文件 → 各自独立的 file 对象 → 各自有独立的读写位置
- **同一个 inode**：无论哪种方式，最终都指向同一个 inode（同一文件），修改互相可见
- **fork 的注意点**：多进程同时写同一文件需谨慎——可使用 `O_APPEND` 保证原子追加

---

## 9.4 页缓存 Page Cache ★★★★★

### 工作原理

Page Cache 是内核在内存中为文件数据维护的缓存，位于文件系统和块设备之间。

```
用户进程
    │
    ▼  read/write
Page Cache（内存） ← 缓存最近读写的文件页面
    │
    ▼  缺页/回写
磁盘（块设备）
```

**核心机制**：
- 第一次读文件 → 从磁盘读入 Page Cache
- 后续读 → 直接命中 Page Cache（内存速度）
- 写文件 → 先写到 Page Cache，标记脏页，异步回写磁盘
- 内存紧张时 → 换出干净的页；脏页先刷盘再换出

### 面试题

**Q: Page Cache 是什么？**

> Page Cache 是 Linux 内核在内存中维护的文件缓存（由物理内存页组成），位于文件系统和磁盘之间。
>
> **作用**：
> 1. **加速读**：第一次读落地，后续读命中缓存（内存速度）
> 2. **缓冲写**：写操作先到 Page Cache 就返回（异步刷盘），提高写性能
> 3. **预读**：内核预测访问模式，提前加载后续数据
>
> **注意**：
> - `free -h` 中 `buff/cache` 就是 Page Cache
> - 可被回收，`echo 3 > /proc/sys/vm/drop_caches` 手动清理
> - 脏页必须刷盘后才能回收
> - 大文件大量随机 I/O 时 Page Cache 效果变差

## 9.5 硬链接与软链接 ★★★★

### 硬链接（Hard Link）

```bash
ln source.txt hardlink.txt   # 创建硬链接
```

- **本质**：多个**目录项**指向**同一个 inode**
- 硬链接使文件的**链接计数** +1
- 删除任何一个硬链接（包括"原文件"），数据仍存在
- 只有当**链接计数降到 0** + 所有进程关闭文件 → inode 和数据块才被释放
- **限制**：不能跨文件系统，不能给目录创建硬链接（避免循环）

### 软链接（Symbolic Link / Symlink）

```bash
ln -s source.txt softlink.txt  # 创建软链接
```

- **本质**：一个独立的文件（有自己的 inode），存储的是**目标文件的路径**
- 软链接类似 Windows 的"快捷方式"
- 可以跨文件系统，可以指向目录
- 原文件删除后，软链接变成"死链接"（dangling link）
- 权限是 lrwxrwxrwx，实际权限取决于目标文件

### 对比

```
硬链接：
  文件名A ─┐
            ├──→ inode 1234 → 数据块
  文件名B ─┘    (链接计数=2)

软链接：
  软链接文件(inode 5678) → 数据块存储"/path/target" → 目标文件(inode 1234)
```

| 维度 | 硬链接 | 软链接 |
|------|--------|--------|
| **inode** | 同目标 inode | 独立的 inode |
| **跨文件系统** | ❌ 不支持 | ✅ 支持 |
| **链接目录** | ❌ 不允许 | ✅ 支持 |
| **目标删除后** | 数据仍在 | 死链接 |
| **磁盘占用** | 仅目录项 | 新文件（含路径） |
| **`ls -l` 显示** | 链接计数增加 | `l` 开头 + `->` 指向 |

### 面试题

**Q: 硬链接和软链接有什么区别？**

> 硬链接指向同一个 inode（同文件多个名字），软链接是独立文件存储目标路径（类似快捷方式）。关键区别：硬链接不能跨文件系统、不能链接目录、删原文件不影响硬链接；软链接都可以，但原文件删除后变成死链接。

## 9.6 Buffer 与 Cache 的区别 ★★★★

### 经典面试题

```bash
$ free -h
              total        used        free      shared  buff/cache   available
Mem:           7.7G        2.1G        1.2G        234M        4.4G        5.1G
Swap:          2.0G        0.0B        2.0G
```

**buffer 和 cache 到底有什么区别？**

### 简明定义

| | Buffer | Cache |
|------|--------|-------|
| **全称** | Buffer Cache | Page Cache |
| **缓冲什么** | **块设备的原始数据**（superblock、inode 元数据） | **文件内容**（文件的数据页） |
| **目的** | 块设备 I/O 加速 | 文件读写加速 |
| **粒度** | 块（扇区） | 页（4KB） |
| **关系** | 加速"写入" | 加速"读取+写入" |

**一句话**：Buffer 缓存文件系统的**元数据**（目录、inode），Cache 缓存**文件内容数据**。

### 深入理解

在现代 Linux 中（2.4+），buffer cache 和 page cache 已经**统一**：

- **Page Cache** = 以页为单位的文件数据缓存 + 匿名页
- **Buffer Cache** = buffer_head 结构，指向 page cache 中的页，提供块级访问

```
应用程序读写文件
    │
    ▼
Page Cache (页级缓存：文件内容)
    │
    ▼
Buffer (块级缓存：元数据、superblock、位图)
    │
    ▼
磁盘
```

### 内存回收

- buffer/cache 可以随时**回收**（给应用程序腾内存）
- `available` = free + 可立即回收的 buff/cache
- 手动回收（仅测试用）：`echo 3 > /proc/sys/vm/drop_caches`

### 面试题

**Q: free 命令中 buffer 和 cache 有什么区别？**

> - **Buffer（缓冲区）**：块设备层面的缓存，缓存文件系统元数据（inode、superblock、目录项），服务于块设备 I/O
> - **Cache（页缓存）**：文件层面的缓存，缓存具体文件内容数据，服务于文件读写
>
> 现代 Linux 中两者已统一为 Page Cache，buffer 是 page cache 的子集。`free -h` 中两者合并显示为 `buff/cache`。
> 两者都是可回收内存，`available` 列才是真正可用的内存估算值。

## 9.7 I/O 调度算法（磁盘调度）★★★

### 为什么需要 I/O 调度

机械硬盘（HDD）的**寻道时间**（磁头移动到目标柱面）是主要性能瓶颈（~几ms），远比数据传输时间长。将多个随机 I/O 请求按柱面位置**重排序**，可以大幅减少磁头来回跳动，提高吞吐量。

> **注意**：SSD 没有机械部件，随机访问延迟一致，不需要复杂调度 → 用 noop。

### 四种经典算法

#### FCFS（先来先服务）

```
请求序列（柱面号）：98, 183, 37, 122, 14, 124, 65, 67
当前磁头：53

执行顺序：53 → 98 → 183 → 37 → 122 → 14 → 124 → 65 → 67
总移动柱面数 = 640

缺点：磁头来回跳动，效率低
```

#### SSTF（最短寻道时间优先）

```
优先处理距当前磁头最近的请求

53 → 65(最近) → 67 → 37 → 14 → 98 → 122 → 124 → 183
总移动 = 236（远好于 FCFS）

缺点：边缘柱面可能"饥饿"（永远等不到服务）
```

#### SCAN（电梯算法）

```
磁头像电梯一样单向扫描，到一端后反向

方向→增加：53 → 65 → 67 → 98 → 122 → 124 → 183 → 然后反向 → 37 → 14
总移动 = ~332

优点：不会饥饿，所有位置都会经过
```

#### C-SCAN（循环扫描）

```
只向一个方向扫描，到末端后直接回到起点（不处理回程请求）

→ 53 → 65 → 67 → 98 → 122 → 124 → 183 → 跳回 0 → 14 → 37
更均匀的等待时间
```

### Linux I/O 调度器

| 调度器 | 特点 | 适用场景 |
|--------|------|---------|
| **noop** | 仅 FIFO，不重排序 | SSD、虚拟机 |
| **CFQ**（完全公平队列） | 每个进程分配 I/O 时间片 | 桌面/通用（已移除，被 BFQ 取代） |
| **Deadline** | 设置读/写截止时间，快到截止的优先 | 数据库、高负载服务器 |
| **bfq**（Budget Fair Queueing） | CFQ 的改进版 | 桌面响应性 |
| **kyber** | 基于延迟目标的调度 | NVMe 高性能存储 |
| **mq-deadline** | 多队列版 Deadline | NVMe SSD 推荐（现代 Linux 默认） |

```bash
# 查看/修改 I/O 调度器
cat /sys/block/sda/queue/scheduler
# [mq-deadline] kyber bfq none
echo mq-deadline > /sys/block/sda/queue/scheduler
```

### 面试题

**Q: 为什么 SSD 不需要复杂的 I/O 调度？**

> HDD 的寻道时间与柱面距离相关（磁头需要物理移动），排序能大幅减少寻道时间。SSD 没有机械部件，所有位置的读取延迟基本一致（~0.1ms），排序不会带来性能提升。**noop**（FIFO）即可，更复杂的调度反而增加 CPU 开销。

# 第十章 I/O 模型 ★★★★★

## 10.1 阻塞 I/O（Blocking I/O）

```c
read(fd, buf, size);  // 数据未就绪 → 进程阻塞直到数据就绪
```

- 最传统的 I/O 模型
- 进程发起 read → 内核准备数据 → 拷贝数据到用户空间 → read 返回
- 等待期间进程被**挂起**（阻塞），CPU 去运行其他进程

---

## 10.2 非阻塞 I/O（Non-blocking I/O）

```c
fcntl(fd, F_SETFL, O_NONBLOCK);  // 设置 fd 为非阻塞
read(fd, buf, size);  // 数据未就绪 → 立即返回 -1，errno=EAGAIN
```

- 轮询直到数据就绪
- 不阻塞进程，但需要不断轮询 → CPU 浪费

---

## 10.3 I/O 多路复用 ★★★★★

用一个线程同时监听**多个文件描述符**的状态，哪个就绪就处理哪个。

```
一个线程
    │
    ▼
select/poll/epoll  ← 监控多个 fd
    │
    ├── fd1 可读 → 处理
    ├── fd3 可写 → 处理
    └── fd5 有新连接 → 处理
```

### select

```c
int select(int nfds, fd_set *readfds, fd_set *writefds,
           fd_set *exceptfds, struct timeval *timeout);
```

| 维度 | 说明 |
|------|------|
| **fd 数量** | 默认 1024（FD_SETSIZE），可修改但不高 |
| **数据结构** | 位图（bitmap），每次传入全部 fd_set |
| **内核行为** | 遍历所有 fd，O(n) |
| **返回信息** | 只告诉"有就绪"，不知道具体是哪个 |
| **拷贝次数** | 每次调用都需要**拷贝**全部 fd_set 到内核 |

### poll

```c
int poll(struct pollfd *fds, nfds_t nfds, int timeout);
```

| 维度 | 说明 |
|------|------|
| **fd 数量** | 无上限（链表） |
| **数据结构** | pollfd 数组 |
| **内核行为** | 遍历所有 fd，O(n) |
| **改进** | 与 select 基本相同，但突破了 1024 限制 |

### epoll

```c
int epoll_create(int size);             // 创建 epoll 实例
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event); // 增删改
int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout); // 等待
```

| 维度 | 说明 |
|------|------|
| **fd 数量** | 无上限（仅受内存限制） |
| **数据结构** | 红黑树 + 就绪链表 |
| **内核行为** | **事件驱动**（fd 就绪时通过回调加入就绪链表），O(1) |
| **拷贝** | `epoll_ctl` 一次拷贝 fd，`epoll_wait` 只拷贝就绪事件 |

### 面试题

**Q: select、poll、epoll 区别？**

> | 维度 | select | poll | epoll |
> |------|--------|------|-------|
> | **最大 fd 数** | 1024（默认） | 无上限 | 无上限 |
> | **数据结构** | bitmap（fd_set） | pollfd 数组 | 红黑树 + 就绪链表 |
> | **复杂度** | O(n) 遍历所有 fd | O(n) 遍历所有 fd | O(1) 只处理就绪 fd |
> | **拷贝** | 每次全量拷贝到内核 | 每次全量拷贝到内核 | epoll_ctl 一次拷贝，epoll_wait 只拷就绪 |
> | **触发方式** | 水平触发 | 水平触发 | 水平触发（LT）+ 边缘触发（ET） |
> | **适用场景** | 连接少且活跃 | 连接少 | 大量连接、大部分空闲 |
>
> **结论**：10K+ 并发连接 → **epoll**。少量连接 → select/poll 也够用。

---

## 10.4 信号驱动 I/O

```c
fcntl(fd, F_SETFL, O_ASYNC);  // 设置异步通知
signal(SIGIO, handler);       // 注册信号处理函数
```

- fd 就绪时 → 内核发送 SIGIO 信号 → 信号处理函数中执行 read
- 非阻塞，比轮询好（信号代替轮询）
- 但处理信号复杂（信号处理函数中做 I/O 不理想）

---

## 10.5 异步 I/O（AIO）

```c
// 真正异步 I/O (Linux AIO / io_uring)
aio_read(&cb);  // 立即返回，内核完成数据准备+拷贝后回调
```

- **与信号驱动 I/O 的区别**：
  - 信号驱动：通知你"可以读了"，数据还没到用户空间
  - 异步 I/O：内核完成读取+拷贝后通知你"数据已经准备好了"
- Linux 原生 AIO 限制多（只支持 O_DIRECT），`io_uring` 是新一代方案

---

## 10.6 五种 I/O 模型总结 ★★★★★

```
                    阻塞     非阻塞   多路复用   信号驱动   异步I/O
                    I/O       I/O       I/O        I/O
                    
发起I/O  ────────────────────────────────────────────
        │ 阻塞       │ 立即返回  │ select/   │ 立即返回  │ 立即返回
        │ 直到完成   │ EAGAIN   │ epoll阻塞 │ 发信号    │
        │            │          │ 等待事件  │          │
                    
等待数据  ───────────│─ 轮询 ───│──────────│──────────│─ 内核负责 ─
        │            │          │           │          │
                    
拷贝数据  ───────────│──────────│─── read ──│─ read ───│─ 内核负责 ─
        │            │          │           │          │
                    
完成通知  进程继续    │ 进程主动  │ 进程主动  │ 信号通知  │ 回调/信号
                     │ 检查      │ 检查      │          │
```

### 面试题

**Q: Linux 五种 I/O 模型是什么？**

> | 模型 | 特点 | 典型 API |
> |------|------|---------|
> | **阻塞 I/O** | 等待期间进程挂起 | read/write（默认） |
> | **非阻塞 I/O** | 轮询检查，不阻塞进程 | read + O_NONBLOCK |
> | **I/O 多路复用** | 一个线程监控多个 fd | select/poll/epoll |
> | **信号驱动 I/O** | fd就绪 → 内核发信号 | SIGIO + fcntl |
> | **异步 I/O** | 内核全包，完成后通知 | aio_read / io_uring |
>
> **前四种都是同步 I/O**（即数据从内核→用户空间的拷贝阶段，进程/线程需要等待或主动执行 read）。
> **只有异步 I/O 是真正的异步**——内核完成所有工作后才通知进程。

## 11.1 epoll 原理

### 红黑树 + 就绪链表

```
epoll_create → 创建 eventpoll 对象
                 ├── rbr（红黑树）：存储所有监控的 fd
                 └── rdllist（就绪链表）：存储已就绪的 fd

epoll_ctl(ADD, fd)
     → 将 fd 插入红黑树
     → 向内核注册回调：fd 就绪时将该节点加入 rdllist

epoll_wait()
     → 检查 rdllist 是否为空
     → 有数据：返回就绪事件列表
     → 无数据：阻塞等待（可设超时）
```

**为什么快？**
1. **事件驱动**：fd 就绪时通过回调加入就绪链表，不需要遍历所有 fd
2. **O(1)**：epoll_wait 只需检查就绪链表，复杂度与 fd 总数无关
3. **一次注册持续有效**：epoll_ctl 注册一次即可，无需每次 wait 时重新传入全部 fd（select/poll 每次都要）
4. **LT/ET 模式**：ET（边缘触发）减少 epoll 通知次数，减少系统调用
5. **活跃就绪时才拷贝**：仅拷贝实际就绪的事件给用户态，而非全部 fd

> **注意**：网上常见"epoll 用 mmap 共享内存减少拷贝"的说法，实际主线 Linux 内核 epoll 并未使用 mmap，`epoll_wait` 通过 `copy_to_user` 拷贝就绪事件。epoll 快的核心原因是事件驱动回调 + O(1) 就绪链表。

### 水平触发 LT vs 边缘触发 ET ★★★★★

这是 epoll 独有的两种工作模式，是面试高频考点。

**水平触发（Level Triggered，LT）**——默认模式：

```
只要 fd 的缓冲区有数据可读，epoll_wait 就会持续通知
→ 这次没读完，下次 epoll_wait 还会通知
→ 类似于"只要锅里还有饭，就一直叫你来吃"
```

```c
// LT 模式（默认）：不一次性读完也没关系，下次还会通知
event.events = EPOLLIN;  // 默认就是 LT
char buf[1024];
int n = read(fd, buf, sizeof(buf));  // 只读了部分数据
// 下次 epoll_wait 如果缓冲区还有数据，会再次通知
```

**边缘触发（Edge Triggered，ET）**——高性能模式：

```
只在 fd 状态变化时通知一次（如：无数据 → 有数据）
→ 必须一次性读完所有数据（读到 EAGAIN）
→ 如果没读完，剩余数据不会再次通知，除非有新数据到达
→ 类似于"只叫你一次，吃没吃完是你的事"
```

```c
// ET 模式：必须循环读到 EAGAIN，否则数据可能丢失
event.events = EPOLLIN | EPOLLET;  // 设置 ET 模式
fcntl(fd, F_SETFL, O_NONBLOCK);    // ET 必须配合非阻塞！

while (1) {
    int n = read(fd, buf, sizeof(buf));
    if (n < 0) {
        if (errno == EAGAIN || errno == EWOULDBLOCK) {
            break;  // 读完了
        }
        // 真正的错误处理
    }
    // 处理 buf 中的数据
}
```

**LT vs ET 对比**：

| 维度 | LT 水平触发 | ET 边缘触发 |
|------|-----------|-----------|
| **通知策略** | 有数据就通知（状态） | 状态变化时才通知一次（边沿） |
| **数据没读完** | 下次还会通知 | **不会**再通知！可能丢数据 |
| **工作方式** | 阻塞/非阻塞均可 | **必须非阻塞** + 循环读到 EAGAIN |
| **编程复杂度** | 简单 | 较高（容易漏数据） |
| **系统调用次数** | 较多（持续通知） | 少（只通知一次） |
| **适用场景** | 简单可靠，兼容性好 | 高性能场景，减少 epoll_wait 调用 |
| **Go 中** | netpoller 默认类似 LT | 可通过 epoll+syscall 手动使用 |

**为什么 ET 必须配合非阻塞 I/O？**

> ET 模式下需要循环 `read` 直到返回 `EAGAIN`。如果 fd 是阻塞模式，最后一次 `read` 会**永久阻塞**在那里。而非阻塞模式在数据读完后立即返回 `EAGAIN`，程序可以退出循环，继续等待 epoll 通知。

**面试题**：

**Q: epoll 的 LT 和 ET 有什么区别？为什么 ET 必须非阻塞？**

> LT（水平触发）只要缓冲区有数据就持续通知，没读完下次还会提醒，安全简单。ET（边缘触发）只在状态变化时通知一次，要求一次性读完到 EAGAIN，通知次数少性能高。
>
> ET 必须非阻塞：因为需要循环读到 EAGAIN，阻塞模式会在最后一次 read 时永远卡住。
>
> **选择**：一般用 LT（默认，够用且安全），极致性能场景用 ET（如 Nginx 处理大量空闲连接）。

## 11.2 Reactor 模型（反应堆）

Reactor 是一种**事件驱动**的编程模型，用于处理并发 I/O。

### 单 Reactor 单线程

```
┌──────────────────────────┐
│         Reactor           │
│  (accept + read + write) │
│         单线程             │
└──────────────────────────┘
          │
    多个 Client
```

- Redis 采用的模型（6.0 之前）
- **优点**：简单，无线程同步问题
- **缺点**：不能利用多核，一个 handler 阻塞会影响所有连接

### 单 Reactor 多线程

```
┌────────────┐
│  Reactor   │ ← 单线程（accept + 分发）
│  (主线程)  │
└─────┬──────┘
      │ 分发事件
      ▼
┌─────────────────┐
│  Thread Pool    │ ← 多个 Worker 线程
│  (read + write) │    (实际处理)
└─────────────────┘
```

- **Reactore 单线程**负责接受连接和分发事件
- **Worker 线程池**处理实际的读写和业务逻辑

### 主从 Reactor（多 Reactor）

```
┌──────────────┐
│ Main Reactor │ ← 只负责 accept 新连接
│   (主线程)   │
└──────┬───────┘
       │ 分发连接
       ▼
┌──────────────┐  ┌──────────────┐
│Sub Reactor 1 │  │Sub Reactor 2 │ ... ← 每个负责一部分连接的 I/O
│  + Worker    │  │  + Worker    │
└──────────────┘  └──────────────┘
```

- **Netty** 和 **Go netpoll** 是这类模型
- Main Reactor 只做 accept，连接分发给 Sub Reactor
- Sub Reactor 负责连接上的所有 I/O 事件

## 11.3 Proactor 模型

与 Reactor 的区别在于**谁来执行实际的 I/O 操作**：

| 模型 | I/O 执行者 | 通知内容 |
|------|-----------|---------|
| **Reactor** | 应用程序自己 read/write | "可以读了" |
| **Proactor** | 操作系统异步完成 read/write | "已经读好了，数据在这" |

Proactor 需要操作系统支持真正的异步 I/O（如 IOCP on Windows, io_uring on Linux）。

## 11.5 C100K 问题

**C100K = 100,000 个并发连接。**

C10K 解决后，挑战变成 C100K：

| 层面 | 瓶颈 | 解决方案 |
|------|------|---------|
| **内核** | epoll 本身处理 100K fd 没问题 | 采用多 Reactor 模型 |
| **内存** | 100K 连接的 socket buffer | 调优 tcp_mem, tcp_rmem |
| **CPU** | 频繁中断、上下文切换 | 多队列网卡 + RSS/RPS + 中断亲和性 |
| **应用层** | 串行处理慢 | 异步非阻塞 + 协程/事件驱动 |
| **文件描述符** | 默认 1024 | `ulimit -n 1000000` |

**关键优化**：

**1. 网卡多队列（RSS）+ 中断亲和性（IRQ Affinity）**

现代万兆/40G 网卡支持**多队列**（Multi-Queue），每个队列有独立的中断号。通过 RSS（Receive Side Scaling），硬件根据数据包的源 IP/端口哈希将其分发到不同队列，使中断均匀分散到多个 CPU 核。

```
传统单队列：                        多队列 + RSS + 中断亲和性：
  所有数据包                            数据包
      │                                    │
      ▼                           RSS 哈希分发 (源IP:Port→Hash)
  队列0 ──► 中断 IRQ42                   │
      │                    ┌──────────────┼──────────────┐
      ▼                    ▼              ▼              ▼
    CPU0                 队列0           队列1           队列2
   (单个 CPU 被           IRQ42          IRQ43          IRQ44
    中断打满)              │              │              │
                          ▼              ▼              ▼
                        CPU0           CPU1           CPU2
                      (中断分散，无单核瓶颈)
```

```bash
# 查看网卡队列数和中断号
ethtool -l eth0                           # 队列数量
cat /proc/interrupts | grep eth0          # 每个队列的中断号和处理 CPU

# 设置中断亲和性（将 IRQ 绑定到特定 CPU 核）
echo 2 > /proc/irq/42/smp_affinity        # IRQ42 → CPU1（0b0010）
echo ff > /proc/irq/42/smp_affinity       # IRQ42 → CPU0-7（8个核都可处理）

# RPS（Receive Packet Steering）—— 软件层模拟 RSS
# 网卡不支持多队列时，由内核软件分发数据包到不同 CPU
echo ff > /sys/class/net/eth0/queues/rx-0/rps_cpus  # 8 个核参与处理
```

**2. 内核旁路（Kernel Bypass）**：DPDK 直接从用户态操作网卡（绕过内核协议栈），将数据包直接映射到用户态内存，消除内核拷贝和系统调用开销。

**3. TCP 参数调优**（tcp_tw_reuse, tcp_fin_timeout, tcp_max_syn_backlog 等）。

## 11.7 Go Netpoll

Go 运行时的网络轮询器（Netpoller）是 Go 高性能网络编程的基础：

```
              ┌─────────────┐
  goroutine → │  network    │ → 非阻塞 I/O → 返回 EAGAIN
              │  poller     │
              └──────┬──────┘
                     │ fd 就绪时通知
                     ▼
              ┌──────────────┐
              │ epoll/kqueue │ (操作系统)
              └──────────────┘
```

**工作原理**：
1. Goroutine 执行网络 I/O 时，Go runtime 将 fd 设为**非阻塞**
2. 如果 I/O 不能立即完成（EAGAIN）→ Netpoller 接管
3. 当前 G 被**挂起**（park），M 继续运行其他 G
4. Netpoller 在后台通过 epoll 监控 fd
5. fd 就绪 → Netpoller 通知 → G 被重新加入调度队列
6. 对用户来说就像"阻塞 I/O 一样简单"，但实际上底层是**异步非阻塞**

**优势**：
- 编程模型简单（看起来像同步阻塞，实际异步非阻塞）
- 无需维护线程池，goroutine 极其廉价
- 天然集成 GMP 调度模型

## 11.8 GMP 调度模型 ★★★★★

Go 的协程调度模型，是 Go 高并发的核心。

### G（Goroutine）

- Go 协程，用户态轻量级线程
- 包含：栈（2KB 起）、程序计数器、调度信息
- 状态：_Gidle, _Grunnable, _Grunning, _Gsyscall, _Gwaiting, _Gdead

### M（Machine / OS Thread）

- 操作系统线程，是 Go 运行时与内核的桥梁
- M 只有拿到 P 才能执行 G
- 默认最大 10000 个 M（`debug.SetMaxThreads`），实际通常很少

### P（Processor）

- 逻辑处理器，是 G 和 M 之间的调度中介
- 每个 P 维护一个**本地 G 队列**（runqueue，最多 256 个 G）
- P 的数量 = `GOMAXPROCS`（默认 CPU 核数）
- P 是 **G 调度的核心**

### GMP 协作流程

```
 ┌─────┐    ┌─────┐    ┌─────┐
 │ G G │    │ G G │    │ G G │  ← 本地队列（runqueue）
 │ G G │    │ G   │    │ G G │
 └──┬──┘    └──┬──┘    └──┬──┘
    P1         P2         P3       ← GOMAXPROCS
    │          │          │
    M1         M2         M3       ← OS Threads (KSE)
    │          │          │
    ▼          ▼          ▼
         CPU Cores
```

**调度流程**：

1. **创建 G**：`go func()` → 新 G 放入当前 P 的本地队列
2. **P 找 G**：P 从本地队列取 G 交给 M 执行
3. **本地队列空**：P 从**全局队列**取 G（加锁）
4. **全局队列也空**：P 从**其他 P 的本地队列偷一半**（Work Stealing）
5. **G 阻塞（系统调用）**：M 与 P 分离，P 绑定新 M → 不浪费 CPU
6. **G 阻塞（网络 I/O）**：G 交给 Netpoller，M 和 P 继续运行其他 G
7. **抢占调度**（Go 1.14+）：基于信号的异步抢占，防止 CPU 密集型 G 霸占 CPU

**关键设计**：
- **Work Stealing**：空闲 P 从其他 P 偷 G，保证负载均衡
- **Hand Off**：M 阻塞时 P 会"分手"找新的 M
- **全局队列**：当本地队列满了（256），新 G 放到全局队列

### 面试题

**Q: GMP 模型是什么？**

> GMP 是 Go 的协程调度模型：
> - **G（Goroutine）**：协程，用户态轻量线程，初始栈 2KB
> - **M（Machine）**：OS 线程，必须绑定 P 才能执行 G
> - **P（Processor）**：逻辑处理器，拥有本地 G 队列，数量 = GOMAXPROCS
>
> **核心机制**：Work Stealing（偷任务）、Hand Off（M 阻塞时 P 转移）、抢占调度。
> M 数量可以远大于 P（系统调用阻塞时会有额外 M），但活跃的 M ≤ P 的数量。

**Q: Goroutine 如何调度？**

> 1. `go func()` 创建 G → 放入当前 P 的本地队列
> 2. P 从本地队列取 G → 交给 M 执行
> 3. 本地队列空 → 从全局队列或偷其他 P 的 G
> 4. 系统调用阻塞 → M 和 P 分离，P 找其他 M 继续运行
> 5. 网络 I/O → G 交给 Netpoller，就绪后被重新调度
> 6. 抢占 → 运行超过 10ms 的 G 被标记为可抢占（Go 1.14+ 异步抢占）

## 11.9 Go GC 机制 ★★★★★

Golang 后端校招必考题。Go 从 1.5 起使用**并发的三色标记-清除**（Concurrent Tri-color Mark-Sweep）GC。

### 三色标记法

所有堆对象被标记为三种颜色：

| 颜色 | 含义 |
|------|------|
| ⬜ **白色** | 尚未被扫描的对象（初始状态，GC 结束时白色的 = 垃圾，可回收） |
| ⬜ **灰色** | 已被扫描但其**引用的子对象**尚未扫描 |
| ⬛ **黑色** | 已被扫描且其引用的子对象也已被扫描 |

```
GC 流程：
1. 初始阶段（STW #1）：
   - 根对象（全局变量、栈变量、寄存器引用）标记为灰色
   
2. 并发标记（并发执行，不 STW）：
   - 从灰色对象出发，扫描其引用的子对象
   - 子对象标记为灰色，自己变为黑色
   - 重复直到没有灰色对象
   
3. 重新扫描（STW #2）：
   - 处理并发标记期间新产生的引用变化
   
4. 并发清除（并发执行）：
   - 回收所有白色对象的内存
```

### 写屏障（Write Barrier）

**为什么需要写屏障？** 并发标记期间，用户代码（mutator）可能在修改对象引用。如果不加保护：

```
并发标记过程中：
  - 黑色对象 A 原本引用白色对象 C → 扫描完成，A 变黑
  - 此时 mutator 将 A.field = nil，灰色对象 B.field = C
  - GC 从灰色 B 继续扫描 → 但 B 已经变灰，不会重新扫 B
  - C 被错误地当作垃圾回收！
```

**解决方案——混合写屏障**（Go 1.8+）：

Go 同时使用 **Dijkstra 插入屏障** + **Yuasa 删除屏障**：

```
Dijkstra 插入写屏障：当写操作将指针 x 赋值给对象的某字段时
  - 将 x 标记为灰色（确保新引用的对象不会被漏扫）
  
Yuasa 删除写屏障：当写操作覆盖对象的某字段时
  - 将旧值标记为灰色（确保被删除引用的对象在下一轮 GC 有机会被扫）
```

**效果**：既不需要完全 STW，也不影响 Go 栈的性能（栈上的操作不需要写屏障，只在堆上）。

### STW（Stop The World）阶段

Go 1.5+ 只需要两次短暂的 STW：

| 阶段 | 耗时 | 说明 |
|------|------|------|
| **初始标记** | ~几十 μs | 标记根对象（全局变量、goroutine 栈） |
| **重新标记** | ~几十~几百 μs | 处理并发标记期间产生的引用变更 |

**并发标记阶段不 STW**，与用户 goroutine 并行执行。整个 GC 周期中，STW 总时间通常 < 1ms。

### GC 触发条件

```go
// GOGC 默认 100，含义：
//   上次 GC 后堆内存增长 100% 时触发下一次 GC
//   如上次 GC 后堆 10MB → 堆到 20MB 时触发下次 GC

// 也可手动设置
debug.SetGCPercent(200)  // 堆增长 200% 才触发 GC（减少 GC 频率）
runtime.GC()             // 手动触发 GC
```

**触发条件**：
1. **GOGC 阈值**：堆增长达到 GOGC 百分比（默认 100%）
2. **时间触发**：距离上次 GC 超过 2 分钟（强制触发）
3. **手动触发**：`runtime.GC()`

### GC 调优与排查

```bash
# 查看 GC 日志
GODEBUG=gctrace=1 ./myapp
# 输出示例：
# gc 1 @0.012s 2%: 0.12+1.3+0.10 ms clock, 0.48+0.5/1.2/0+0.40 ms cpu
#        │        │     └─ STW 时间
#        │        └─ GC 阶段的 CPU 时间占比
#        └─ GC 次数

# pprof 排查 GC 问题
curl http://localhost:6060/debug/pprof/heap   # 堆内存分布
curl http://localhost:6060/debug/pprof/allocs # 内存分配热点
go tool trace trace.out                       # 可视化 GC 事件时间线
```

**减小 GC 压力的常见手段**：
1. 减少堆上的对象数（用 slice/array 代替链表）
2. sync.Pool 复用频繁分配的对象
3. 避免不必要的指针和 interface{} 装箱（减少逃逸分析压力）
4. 字符串拼接用 `strings.Builder` 而非 `+=`
5. 调大 GOGC（如 200、500）减少 GC 频率（以更多内存为代价）

### 面试题

**Q: Go GC 是什么原理？三色标记法如何工作？**

> Go 使用**并发的三色标记-清除** GC。对象被标为白（未扫）、灰（正在扫）、黑（已扫）三种颜色。从根对象出发，灰色对象的子对象变灰，自己变黑，直到没有灰色对象 → 清除白色（垃圾）。
>
> **并发安全**：通过混合写屏障（Dijkstra + Yuasa）保证并发标记期间 mutator 修改引用不会导致漏标。
>
> **STW**：只在初始标记和重新标记两个阶段短暂暂停（通常 < 1ms），标记阶段与用户代码并发执行。
>
> **触发**：GOGC（默认 100）控制，堆增长 100% 触发下次 GC。

**Q: Go GC 的写屏障是什么？为什么需要？**

> 写屏障是并发 GC 中保证正确性的关键机制。并发标记期间，用户代码（mutator）可能修改对象引用，导致已扫描的黑色对象重新引用白色对象，使白色对象被漏标（错误回收）。
>
> Go 1.8+ 使用混合写屏障：
> - **插入屏障（Dijkstra）**：新引用的对象标记为灰色
> - **删除屏障（Yuasa）**：被删除引用的对象标记为灰色
>
> 栈上的指针操作不需要写屏障优化（因为 Go 栈在 GC 期间会被扫描），只在堆对象上使用。

## 12.6 常见排查流程

### CPU 飙高如何排查？

```
1. top / htop                    → 找到 CPU 最高的进程 PID
2. top -Hp <pid>                → 找到进程中最耗 CPU 的线程 TID
3. printf "%x\n" <tid>          → 将 TID 转为 16 进制
4. jstack <pid> | grep <nid>    → (Java) 找到对应线程的堆栈
   OR perf top -p <pid>         → (C/C++) 找到热点函数
   OR curl localhost:<debug_port>/debug/pprof/profile?seconds=30  → (Go) pprof 采样
5. 分析堆栈 → 定位具体代码行
```

**Go 专属流程**：
```bash
# 1. pprof 采样 CPU
curl http://localhost:6060/debug/pprof/profile?seconds=30 > cpu.prof
go tool pprof cpu.prof
# (pprof) top  → 看最耗时的函数
# (pprof) list funcName → 看具体代码行

# 2. 或者 kill -SIGQUIT <pid> 让 Go 进程 dump 所有 goroutine 堆栈
# 查看 goroutine 卡在哪里
```

### 内存泄漏如何排查？

```
1. top / htop              → 观察 RES/VIRT 是否持续增长
2. pmap -x <pid>           → 查看内存映射分布
3. cat /proc/<pid>/smaps   → 详细内存区域分析
4. valgrind --leak-check=full ./app  → (C/C++) 内存泄漏检测
5. 或使用 AddressSanitizer 编译 (-fsanitize=address)
```

**Go 专属流程**：
```bash
# 1. pprof 采样堆内存
curl http://localhost:6060/debug/pprof/heap > heap.prof
go tool pprof heap.prof
# (pprof) top  → 看哪个函数分配内存最多

# 2. 对比两次 heap profile
go tool pprof -base heap1.prof heap2.prof  → 差分看增长

# 3. goroutine 泄漏检查
curl http://localhost:6060/debug/pprof/goroutine > goroutine.prof
# 如果 goroutine 数量持续增长 → goroutine 泄漏
```

## 12.7 Load Average 详解 ★★★★

### 三个数字的含义

```bash
$ uptime
12:34:56 up 10 days, 3:14, 5 users, load average: 0.45, 0.72, 0.89
                                                     1min   5min   15min
```

这三个数字分别表示过去 **1分钟、5分钟、15分钟**的平均负载。

### Load Average 究竟是什么？

**Load Average = 处于"可运行状态"（R状态）+ "不可中断睡眠"（D状态）的进程数的指数移动平均值。**

```
可运行状态 (R/TASK_RUNNING): 正在运行 + 在就绪队列等待 CPU
不可中断睡眠 (D/TASK_UNINTERRUPTIBLE): 正在等待 I/O（通常是磁盘 I/O）
```

**注意**：Load Average **不是 CPU 使用率**！它包含了等待 I/O 的进程。

### 如何解读

```
单核系统：
  load < 0.7  → 轻松
  load ~ 1.0  → 刚好满载
  load > 1.0  → 过载，有进程在排队等待

多核系统：
  load = CPU核心数 → 刚好满载（所有核都有活干，没有排队）
  load > CPU核心数 → 过载
  load = 4.0  在 4核机器上 → 满载
  load = 4.0  在 2核机器上 → 严重过载
```

**实例分析**：

| 场景 | top 显示 | Load Avg | 原因 |
|------|---------|----------|------|
| CPU 密集 | %us 高 | 升高 | 进程在 R 状态排队 |
| 磁盘 I/O 密集 | %wa 高 | 升高 | 进程在 D 状态等待磁盘 |
| 大量进程等待 | 正常 | 飙升 | 大量进程在就绪队列 |

**CPU 高但 load 不高**：少数几个进程死循环但核心数足够（如 1 核满 7 核空闲）
**Load 高但 CPU 不高**：大量进程等待 I/O（D 状态）→ 磁盘是瓶颈

### 面试题

**Q: Load Average 的三个数字是什么含义？CPU 不高但 Load 高是为什么？**

> Load Average 是 1/5/15 分钟的平均系统负载，统计的是 **R 状态（运行/等待CPU）** + **D 状态（不可中断睡眠/等待I/O）** 的进程数。
>
> **CPU 不高但 Load 高**：说明大量进程卡在 **D 状态**（等待磁盘 I/O），CPU 在空转等待数据，但不是 CPU 瓶颈。典型场景：大量并发读写慢磁盘、NFS 挂载卡住、数据库大量随机 I/O。
>
> **排查**：`iostat -x 1` 看磁盘 %util 和 await；`ps aux` 看 D 状态进程；`cat /proc/<pid>/stack` 看卡在内核哪个函数。

## 12.8 守护进程 Daemon ★★★

### 什么是守护进程

守护进程（Daemon）是在后台运行的**常驻进程**，独立于控制终端，周期性执行某些任务或等待处理某些事件。

**特征**：
- 没有控制终端（tty 显示为 `?`）
- 父进程通常是 init/systemd（PID=1）
- 进程名通常以 `d` 结尾（如 `sshd`, `nginx`（master）、`mysqld`, `crond`）

### 创建守护进程的步骤

```c
// 1. fork 子进程，退出父进程（使子进程成为孤儿，被 init 收养）
pid = fork();
if (pid > 0) exit(0);

// 2. 子进程创建新会话（脱离原终端）
setsid();

// 3. 再次 fork，退出第一代子进程（防止获得控制终端）
pid = fork();
if (pid > 0) exit(0);

// 4. 切换工作目录到根目录（避免占用目录导致无法卸载）
chdir("/");

// 5. 重设文件权限掩码
umask(0);

// 6. 关闭继承的文件描述符（stdin/stdout/stderr 等）
close(0); close(1); close(2);
// 重定向到 /dev/null
open("/dev/null", O_RDWR);  // fd=0
dup(0);                     // fd=1
dup(0);                     // fd=2

// 7. 执行守护进程的实际工作
while (1) {
    // do something
    sleep(interval);
}
```

**Go 中创建守护进程**：Go 不推荐直接写 daemon，通常由 systemd/supervisor 等进程管理工具托管。

## 12.9 cgroups 与 Namespace ★★★★★

Docker/K8s 容器的两大底层技术，校招高频考点。

### cgroups（Control Groups）— 资源限制

cgroups 是 Linux 内核的**资源隔离和限制**机制，可以限制、记录、隔离进程组的物理资源。

**四种资源控制（v1）：**

| 子系统 | 控制内容 | 示例限制 |
|--------|---------|---------|
| **cpu** | CPU 使用时间 | `--cpus=2` 限制容器的 CPU 核心数 |
| **memory** | 内存使用上限 | `--memory=512M` 超出则 OOM Kill |
| **blkio** | 磁盘 I/O 带宽 | 限制容器读写磁盘的 IOPS/BPS |
| **pids** | 最大进程数 | 防止 fork 炸弹 |

**cgroups v2**（推荐）：单一层级统一管理所有资源，v1 已被逐步淘汰。

```bash
# 查看进程所属的 cgroup
cat /proc/<pid>/cgroup

# Docker 中限制资源（底层就是 cgroups）
docker run --cpus=2 --memory=512m myapp
```

**与 OOM Killer 的关系**：容器内内存超出 cgroup 限制 → 触发 cgroup 级别的 OOM Killer（先于系统级 OOM）。`oom_score_adj` 在容器中同样生效。

### Namespace — 视图隔离

Namespace 让每个进程组**只能看到属于自己的系统资源视图**，实现进程间隔离。

| Namespace | 隔离内容 | 效果 |
|-----------|---------|------|
| **PID** | 进程 ID | 容器内进程 PID=1，看不到宿主机其他进程 |
| **Mount** | 文件系统挂载点 | 容器有独立的 `/` 根文件系统 |
| **Network** | 网络栈 | 容器有独立的网卡、IP、端口、路由表 |
| **UTS** | 主机名和域名 | 容器有独立 hostname |
| **IPC** | 信号量/消息队列/共享内存 | 容器内 IPC 资源隔离 |
| **User** | 用户和组 ID | 容器内 root ≠ 宿主机 root |
| **Cgroup** | cgroup 视图 | 容器只能看到自己的 cgroup 限制 |

```bash
# 查看进程的 namespace
ls -l /proc/<pid>/ns/
```

### Docker 如何利用两者

```
Docker 容器 = cgroups（限制资源）+ Namespace（隔离视图）+ UnionFS（分层镜像）
```

- **docker run --memory** → cgroups memory 子系统
- **docker run --cpus** → cgroups cpu 子系统
- **容器内 ps 只能看到自己** → PID namespace
- **容器有独立 IP** → Network namespace

### 面试题

**Q: Docker 底层用了什么 Linux 技术？**

> 三大技术：
> 1. **cgroups**（Control Groups）：限制容器的 CPU、内存、磁盘 I/O 等资源使用上限
> 2. **Namespace**：实现进程隔离，让容器只能看到自己的 PID、网络、文件系统、用户等（共 7 种）
> 3. **UnionFS**（OverlayFS/aufs）：分层镜像，实现容器镜像的增量构建和共享
>
> **一句话**：cgroups 做"限额"，Namespace 做"隔离"，UnionFS 做"镜像分层"。

# 第十三章 后端面试场景题 ★★★★★

---

## 场景1：进程和线程区别？

> **一句话**：进程是资源分配的基本单位，线程是 CPU 调度的基本单位。
>
> **展开**：
> - 进程独享地址空间和资源，线程共享进程的资源
> - 进程创建/切换开销大（切换页表），线程开销小
> - 进程间通信需要 IPC（管道、共享内存等），线程直接读写共享变量
> - 进程间相互隔离更安全，一个线程崩溃可能影响整个进程

---

## 场景2：协程和线程区别？

> - **调度**：协程由用户态调度（Go runtime），线程由内核调度
> - **栈大小**：协程初始 2KB 动态伸缩，线程固定 1MB~8MB
> - **切换开销**：协程 ~100ns（用户态），线程 ~1-10μs（内核态）
> - **数量**：协程可以创建数十万个，线程受内存限制（几千个）
> - **阻塞影响**：Goroutine 阻塞不影响其他 G，线程阻塞浪费资源
>
> Go 的 Goroutine 走 M:N 混合调度，M 个协程映射到 N 个线程。

---

## 场景3：死锁产生条件？

> 四个必要条件（缺一不可）：
> 1. **互斥**：资源只能被一个进程/线程持有
> 2. **请求保持**：已持有资源又请求新资源
> 3. **不可剥夺**：已获资源不能被强制抢走
> 4. **循环等待**：形成循环等待链

---

## 场景4：如何避免死锁？

> 破坏四个条件之一：
> 1. **按顺序加锁**（破坏循环等待）← 最简单实用
> 2. **tryLock + 超时**（破坏请求保持）
> 3. **一次性申请所有锁**
> 4. **减少锁的粒度/范围**
> 5. 使用无锁数据结构（破坏互斥）
>
> Go 中特别注意：**channel 操作不要成环**。

---

## 场景5：互斥锁和自旋锁区别？

> | 互斥锁 | 自旋锁 |
> |--------|--------|
> | 等待时**阻塞**让出 CPU | 等待时**忙等**占着 CPU |
> | 适合临界区较长 | 适合临界区极短（< 切换开销） |
> | 有上下文切换开销 | 无上下文切换（但浪费 CPU） |
> | 用户态可睡眠 | 中断上下文也可用 |
>
> **简单记忆**：临界区几条指令 → 自旋锁；临界区代码较多 → 互斥锁。

---

## 场景6：CAS 原理是什么？

> CAS（Compare And Swap）是 CPU 的原子指令，操作是：比较内存值是否等于预期值，相等则替换为新值，整个过程是**原子的**。
>
> **ABA 问题**：值从 A→B→A 后 CAS 检测不到。解决方案：加**版本号**（Double-Word CAS）。
>
> **应用**：Go `sync/atomic`、自旋锁、无锁队列、乐观锁。

---

## 场景7：虚拟内存是什么？

> 操作系统为每个进程提供虚拟地址空间的抽象。进程看到的是连续完整的地址空间（4GB/128TB），实际物理内存分散且有限。CPU 通过 MMU（内存管理单元）+ 页表将虚拟地址转换为物理地址。
>
> **价值**：进程隔离、内存保护、内存超分、简化编程。

---

## 场景8：虚拟地址如何转换为物理地址？

> 1. 虚拟地址 = 虚拟页号（VPN）+ 页内偏移
> 2. 先查 **TLB**（快表），命中 → 直接得物理页框号
> 3. TLB Miss → 查多级页表（CR3 → PGD → PUD → PMD → PTE）
> 4. 获取物理页框号 PFN
> 5. 物理地址 = PFN + 页内偏移
> 6. Present=0 → 缺页异常 → 从磁盘换入

---

## 场景9：为什么需要多级页表？

> 单级页表每个进程 4MB（32 位），N 个进程 = N×4MB → 内存爆炸。
>
> 多级页表只分配**实际使用**的虚拟地址区域对应的页表。大部分虚拟地址空间是空洞（不分配页表），大幅节省内存。
>
> 代价：地址转换需要多次访存 → 用 TLB 补偿。

---

## 场景10：TLB 是什么？

> TLB（Translation Lookaside Buffer，快表/页表缓存）是 MMU 内部的高速硬件缓存，缓存最近使用的虚拟页号→物理页框号映射。
>
> TLB 命中 → 地址转换只需 1 个时钟周期。
> TLB Miss → 需要多次访存查多级页表（4 级页表需 4 次）。
>
> 进程切换时 TLB 被刷新（潜在性能杀手）。PCID/ASID 技术可部分缓解。

---

## 场景11：缺页异常发生了什么？

> Page Fault 分三种类型：
> - **Minor**：页已在内存，只需建映射（COW、mmap 延迟加载），不读磁盘，极快
> - **Major**：页不在内存，需从磁盘读入（swap/文件 I/O），慢（ms 级）
> - **Invalid**：非法地址 → SIGSEGV（段错误）
>
> Major Fault 处理：1) 陷入内核 → 2) 检查地址合法性 → 3) 分配物理页框（可能需淘汰其他页）→ 4) 从磁盘 I/O 读入数据 → 5) 更新页表 → 6) 返回用户态重试 → 对进程**透明**。

---

## 场景12：malloc 底层如何实现？

> glibc malloc 根据大小选择策略：
> - **< 128KB**：`brk/sbrk` 移动堆顶指针，释放后内存缓存在内存池
> - **≥ 128KB**：`mmap` 分配独立内存区域，释放后 `munmap` 立即归还 OS
>
> 内部用 **bins/fastbins** 管理空闲 chunk（类似 SLAB），小对象走线程缓存（tcache），减少锁竞争。
>
> Go 使用基于 TCMalloc 思想的内存分配器（mcache → mcentral → mheap 三级缓存）。

---

## 场景13：mmap 为什么快？

> 1. 减少 CPU 拷贝：传统 read+write 有内核→用户的拷贝，mmap 直接操作 Page Cache
> 2. 零系统调用：文件映射后随机访问像操作内存，不需要每次 read/write
> 3. 操作系统透明管理 I/O：缺页自动加载，脏页自动回写
>
> **但不适用于所有场景**：小文件、顺序读写、频繁缺页时 TLB 压力大的场景可能不如 read/write。

---

## 场景14：什么是零拷贝？

> 零拷贝 = **CPU 不参与数据拷贝**（全部由 DMA 完成）。
>
> 传统 read+write：4 次拷贝（2 CPU + 2 DMA），4 次上下文切换。
>
> sendfile + DMA Gather（Linux 2.4+）：2 次 DMA，0 次 CPU 拷贝，2 次上下文切换。
>
> 实现方式：mmap + write、sendfile、splice。
> 应用：Kafka（sendfile 传输）、Nginx（sendfile）、Netty（FileRegion）。

---

## 场景15：Page Cache 是什么？

> Page Cache 是内核在内存中为文件数据维护的缓存。
>
> **作用**：
> - **读缓存**：第一次从磁盘读入，后续命中缓存（内存速度）
> - **写缓冲**：write 先写缓存再异步刷盘（提高写性能）
> - **预读**：内核预测访问模式提前加载
>
> `free -h` 中的 buff/cache 就是 Page Cache。可被回收，但脏页需先刷盘。

---

## 场景16：select、poll、epoll 区别？

> | | select | poll | epoll |
> |------|--------|------|-------|
> | fd 上限 | 1024（默认） | 无上限 | 无上限 |
> | 数据结构 | bitmap | pollfd 数组 | 红黑树+就绪链表 |
> | 复杂度 | O(n) 遍历 | O(n) 遍历 | O(1) 事件驱动 |
> | 拷贝 | 每次全量拷贝 | 每次全量拷贝 | 一次注册 |
> | 触发模式 | LT | LT | LT + ET |
>
> **epoll 为什么快**：事件驱动（回调）、O(1) 就绪链表、一次注册持续有效、只返回就绪 fd。

---

## 场景17：epoll 为什么快？

> 1. **事件驱动**：fd 就绪时通过回调加入就绪链表 → epoll_wait 只处理已就绪的 fd → O(1)
> 2. **红黑树存储**：所有监控的 fd 存在红黑树中，增删改 O(log n)
> 3. **就绪链表**：就绪事件通过回调加入链表，epoll_wait 直接取
> 4. **ET 模式**：边缘触发减少重复通知
> 5. **一次注册持续有效**：epoll_ctl 注册后无需每次传入全部 fd
>
> select/poll 每次都 O(n) 遍历所有 fd → n 大时性能差距巨大。
>
> **注意**：网上误传"epoll 用 mmap 共享内存"，主线 Linux epoll 并未使用 mmap，而是通过 `copy_to_user` 拷贝就绪事件。

---

## 场景18：Reactor 模型是什么？

> Reactor 是**事件驱动**的并发 I/O 处理模型。
>
> - **单 Reactor 单线程**：Redis 6.0 前（简单，不能利用多核）
> - **单 Reactor 多线程**：Reactor 单线程分发 + Worker 线程池处理
> - **主从 Reactor**：Netty、Go Netpoll。Main Reactor accept，Sub Reactor 处理 I/O
>
> 核心思想：I/O 多路复用 + 事件分发 + 非阻塞处理。

---

## 场景19：GMP 模型是什么？

> Go 的协程调度模型，G（Goroutine协程）+ M（OS线程）+ P（逻辑处理器）。
>
> **关键点**：
> - P 数量 = GOMAXPROCS（默认 CPU 核数），每个 P 有本地 G 队列
> - M 必须绑定 P 才能执行 G
> - **Work Stealing**：空闲 P 从其他 P 偷 G
> - **Hand Off**：M 阻塞时 P 找新 M
> - M:N 混合调度（M 个 G 映射到 N 个 OS 线程）
>
> Goroutine 初始栈 2KB，用户态切换 ~100ns，可创建数十万个。

---

## 场景20：CPU 100% 如何排查？

> 1. `top` → 找到高 CPU 的进程 PID
> 2. `top -Hp <pid>` → 找到高 CPU 线程
> 3. `perf top -p <pid>` → 定位热点函数
> 4. **Go**：`pprof` 采样 CPU profile → `go tool pprof` → top/list 定位代码行
> 5. **Java**：`jstack` + 16进制 TID 定位线程堆栈
> 6. 常见原因：死循环、频繁 GC、正则灾难、CAS 自旋、大量系统调用

---

## 场景21：fork() 为什么快？COW 是什么？

> fork 创建子进程时不复制物理内存，只复制页表。父子进程共享的物理页都标记为**只读**——这就是**写时复制（COW）**。
>
> 当任一进程尝试写入时 → 触发页保护异常 → 内核真复制该页（4KB）→ 修改页表指向新页 → 恢复进程。
>
> 效果：fork 创建的只是页表副本（几 KB~几 MB），而非整个地址空间（GB 级），所以极快。很多页面（如 .text 代码段）永远不会被写入，完全省去拷贝。

---

## 场景22：什么是僵尸进程？孤儿进程呢？

> **僵尸进程**：子进程终止后，父进程没有 `wait()` 回收，PCB 残留在系统中。不占内存但**占用 PID**，大量僵尸会导致 PID 耗尽，无法创建新进程。
>
> **孤儿进程**：父进程先于子进程终止，子进程变成孤儿 → 被 init(PID=1) 收养并负责回收。**孤儿进程无害**。
>
> 解决僵尸：父进程调用 `waitpid()` / 忽略 `SIGCHLD` / kill 父进程（让 init 接管）。

---

## 场景23：伪共享是什么？如何解决？

> 伪共享（False Sharing）是多核 CPU 中，不同线程修改**不同变量**，但这些变量在**同一个 Cache Line（64字节）**中。CPU 缓存一致性协议导致该行在核间反复失效/同步，产生显著的性能损耗。
>
> **解决**：缓存行填充（Cache Line Padding），让热变量独占一个 Cache Line。Go 中在结构体字段前后加 `[56]byte` 作为 padding。
>
> 典型场景：高并发计数器的多个 counter、无锁队列的 head/tail 指针。

---

## 场景24：Futex 机制是什么？

> Futex（Fast Userspace muTEX）是 Linux 提供的**混合锁**机制：
> - **无竞争**：纯用户态 CAS 操作（~几十 ns）
> - **有竞争**：通过 `futex()` 系统调用陷入内核，阻塞线程进入等待队列（~几 μs）
>
> 它是 `pthread_mutex`、Go `sync.Mutex`、Java `ReentrantLock` 的底层基础。核心思想：**将内核开销推迟到真正需要的时候**。

---

## 场景25：OOM Killer 是什么？如何保护关键进程？

> OOM Killer 是 Linux 在物理内存 + swap 都耗尽时，强制杀死进程来释放内存的机制。选择 `oom_score` 最高的进程（内存使用量大 + 调整因子）。
>
> **保护关键进程**：
> ```bash
> echo -1000 > /proc/<pid>/oom_score_adj   # 永不被杀
> ```
> 生产环境预防：预留 swap、用 cgroup 限制容器内存、监控内存趋势。

---

## 场景26：硬链接和软链接区别？

> | 硬链接 | 软链接 |
> |--------|--------|
> | 同 inode（同文件多名字） | 独立文件存储目标路径 |
> | 不能跨文件系统 | 可以跨文件系统 |
> | 不能链接目录 | 可以链接目录 |
> | 原文件删除后数据仍在 | 原文件删除后变死链接 |
>
> `ln file hardlink` vs `ln -s file softlink`

---

## 场景27：free 命令中 buffer 和 cache 区别？

> - **Buffer**：块设备层的缓存，存储文件系统**元数据**（inode、superblock、目录项）
> - **Cache（Page Cache）**：文件层的缓存，存储**文件数据内容**
>
> 两者都可回收（`available` 列估算真正可用内存）。
> 现代 Linux 中两者统一在 Page Cache 框架下，buffer 是 page cache 的子集。

---

## 场景28：Load Average 高但 CPU 不高是什么原因？

> Load Average 统计的是 R 状态（运行/等待CPU）+ D 状态（不可中断睡眠/等待I/O）的进程数。
>
> **CPU 不高但 Load 高** → D 状态进程多 → **磁盘 I/O 瓶颈**。进程卡在等待磁盘，CPU 空闲但进程在排队。
>
> 排查：`iostat -x 1` 看 %util 和 await；`ps aux` 看 D 状态进程；`dmesg` 看是否有存储错误。

---

## 场景29：如何创建守护进程？

> 标准步骤（双 fork）：
> 1. `fork()` + 父进程退出 → 子变孤儿，被 init 接管
> 2. `setsid()` → 创建新会话，脱离控制终端
> 3. 再次 `fork()` + 父退出 → 确保不能再获得终端
> 4. `chdir("/")` → 避免占用目录
> 5. `umask(0)` → 重设权限掩码
> 6. 关闭 0/1/2 fd → 重定向到 /dev/null
> 7. 执行实际工作逻辑
>
> Go 中不推荐自己写 daemon，建议用 systemd/supervisor 托管。

---

## 场景30：Linux 内核内存分配层次？

> ```
> 用户态 malloc/free  →  ptmalloc / jemalloc / tcmalloc
> ──────────────────────────────────────────────────
> 内核小对象 kmalloc  →  Slab/SLUB 分配器
> 内核页级 alloc_page →  伙伴系统 Buddy（2^N 页为单位）
> ──────────────────────────────────────────────────
> 物理硬件              →  物理内存页框
> ```
>
> 伙伴系统解决**外部碎片**（分配/释放时合并），Slab 解决**内部碎片**（整页细分小对象）。
